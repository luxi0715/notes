## 第一部分:数据世界 —— 以"数据管线"为锚点

### 1.1 数据的锚点:不是只有表格

绝大多数中国软件工程师有一个共同的认知盲区——**他们工作中接触的数据只有一种:结构化数据**(能放进 MySQL 表里的那种)。

国内 IT 培训和大学课程教的几乎全是这个:Java / Python + Spring / Django + MySQL / Redis + REST API + 增删改查。这套技能完整覆盖了 80% 的传统互联网业务,**但它只覆盖了"数据"这个世界的一个小角落**。

数据的真实锚点是**数据管线**——从产生 → 采集 → 传输 → 处理 → 存储 → 消费的完整流动。流动的是各种模态的数据,不只是表格。

### 1.2 数据类型的全景图

真实世界的数据其实是这十几大类:

| 类型 | 例子 | 存储 | 处理工具 |
|------|------|------|----------|
| **结构化数据** | 订单、用户、商品 | MySQL / PostgreSQL | SQL |
| **半结构化数据** | JSON 日志、配置 | MongoDB / ES | 文档查询 |
| **时序数据** | 监控指标、IoT 传感器、股价 | InfluxDB / TimescaleDB | 时序查询 |
| **图数据** | 社交关系、知识图谱、风控 | Neo4j / NebulaGraph | Cypher |
| **文本** | 文章、评论、聊天记录 | Elasticsearch / 向量库 | 全文检索 / Embedding |
| **图像** | 照片、扫描件、医学影像 | 对象存储 + 特征库 | CV 模型 |
| **音频** | 语音、音乐 | 对象存储 | ASR / 音频指纹 |
| **视频** | 监控、直播、短视频 | 专用存储 + CDN | 抽帧 + CV |
| **3D / 点云** | 自动驾驶、AR/VR、机器人 | 专用格式 | 3D 算法 |
| **生物信号** | 脑电、心电、基因 | HDF5 / 专业格式 | 信号处理 |
| **地理空间** | 位置、轨迹、地图 | PostGIS / 专门数据库 | 空间索引 |

**结构化表格数据,只是这十几类里的一种**。

#### 一个底层规律:数据类型 = 人类感知和测量世界的方式

数据类型的数量,本质上等于人类感知和测量世界的方式数量:

- 人有眼睛 → 图像、视频、3D
- 人有耳朵 → 音频、语音、音乐
- 人有触觉 → 压力、振动、力反馈
- 人会说话 → 文本、对话
- 人有空间感 → 地理、轨迹
- 人能感知时间 → 时序数据
- 人能理解关系 → 图数据(社交、知识)
- 人能测量身体 → 生物信号(心电、脑电、基因)
- 机器能测人测不到的 → 红外、激光雷达、毫米波、磁场、辐射

每一种"感知 / 测量"对应一种数据。所以数据类型的数量,跟随着人类技术能力的扩张而扩张——以前没有激光雷达就没有点云数据,以前没有可穿戴设备就没有连续心率数据。

#### 每一类下面还有十几个细分

光是"图像"这一类,其实包含了:

- 普通 RGB 照片(手机、相机)
- 黑白图像(老照片、文档)
- 红外图像(夜视、热成像)
- 深度图(iPhone 的人像模式、Kinect)
- 多光谱 / 高光谱图像(卫星遥感、农业、矿产勘探)
- 医学影像(X 光、CT、MRI、超声、病理切片)——每一种都是独立的子领域
- 显微图像(生物、材料)
- 雷达图像(SAR 合成孔径雷达)
- 文档图像(扫描件、票据,需要 OCR)
- 工业检测图像(生产线、PCB、芯片)

**光"图像"一项,深耕任何一个子方向就够一个工程师吃一辈子饭**。医学影像的人不懂工业检测,做卫星遥感的不碰文档 OCR——都是图像,但完全是不同的世界。

时序数据也是一样:股票行情、IoT 传感器、服务器监控、心电信号、地震波、音频本身……每一种都有自己的处理方法和工具栈。

### 1.3 数据层级 L1-L9

我用一个核心问题串起来:**"你能让数据创造多大的价值?"**

#### L1 - 单一数据存取
能用一种数据库 CRUD,知道怎么建表、写 SQL、做基本的 join。
**自测**:能写出"查询某用户最近 30 天订单总额"的 SQL 吗?
**对标**:**应届 / 0-1 年**。培训班毕业基本到这层。

#### L2 - 多种存储选型
知道什么场景用什么存储:MySQL / Redis / MongoDB / ES / 对象存储,各自适合什么。
**自测**:用户的购物车数据放 MySQL 还是 Redis?为什么?商品搜索为什么不用 MySQL 用 ES?
**对标**:**初中级,1-3 年**。开始能独立设计一个完整功能的存储方案。

#### L3 - 多模态数据接入
能处理结构化以外的数据——图片、音频、视频、时序、文本、向量——并把它们和业务结合。
**自测**:用户上传一张商品图,你怎么让系统"理解"它(提取标签、找相似商品、过滤违规内容)?一段语音消息怎么转成可搜索的文本?
**对标**:**中级,3-5 年**,而且偏向 AI 时代的工程师。**目前国内大部分工程师卡在 L2 到 L3 之间——这就是"只关注表格数据"盲区的本质**。突破这一层是 2026 年最关键的成长动作。

#### L4 - 数据流与处理管线
能设计完整的数据流转链路:从产生 → 采集 → 传输 → 处理 → 存储 → 消费。会用 Kafka / Flink / Spark / Airflow 这些工具,知道实时流和批处理的区别。
**自测**:每天 1 亿条用户行为日志,怎么实时计算"过去 5 分钟最热的 10 个商品"?数据延迟、丢失、重复怎么处理?
**对标**:**高级,4-7 年**。"数据工程师 / 大数据工程师"的核心岗位职责。**国内大厂 P6-P7、字节 2-2 / 3-1**。

#### L5 - 数据建模与平台架构
能设计一个公司级的数据架构:数据湖 vs 数据仓库 vs 湖仓一体,事实表 / 维度表怎么建,数据血缘 / 数据质量 / 数据治理怎么做。
**自测**:公司有 30 个业务系统、数据散在各处,怎么统一?新业务上线如何一周内接入数据体系?数据出错了怎么追溯?谁能看哪些数据(权限)?
**对标**:**资深,7-10 年**。数据架构师 / 数据平台 Tech Lead。**国内大厂 P7-P8**。

#### L6 - 数据驱动业务
不再只是"存好数据",而是能用数据**改变业务决策和产品形态**——埋点体系、AB 实验平台、用户画像、推荐系统、增长分析。
**自测**:产品要做一个新功能,怎么设计埋点 + 实验 + 数据闭环,让一个月后能用数据证明它是好是坏?用户流失率上升,怎么用数据找原因?
**对标**:**资深 / 专家,8-12 年**。这一层开始要懂业务、懂统计、懂产品。**字节、Meta、Google 这种"数据驱动型公司"的核心人才**。

#### L7 - 智能化:AI + 数据
能把数据变成 AI 能力:特征工程、模型训练数据管线、向量数据库、RAG 系统、推荐 / 搜索 / 风控的算法工程。
**自测**:怎么把公司 10 年的客服对话数据,变成一个能上线的 AI 客服?怎么设计一套 RAG 系统让它的回答质量持续提升?推荐系统的特征怎么设计才不会"信息茧房"?
**对标**:**算法工程 / AI 工程师,5-15 年**(这个赛道年限不重要,能力曲线很陡)。**这是 2026 年最稀缺的方向**。

#### L8 - 数据战略与组织
公司层面的数据战略:数据怎么变成核心竞争力 / 资产 / 商业模式?数据合规(GDPR、个保法、数据出境)怎么做?数据中台值不值得建?自研 vs 用云厂商?
**对标**:**首席数据官 / VP of Data / CTO**。

#### L9 - 行业级影响
定义"数据系统下一代长什么样"的人:Hadoop 的 Doug Cutting、Spark 的 Matei Zaharia、Snowflake 创始团队、向量数据库 Pinecone / Milvus 的核心团队、Databricks 的 Ali Ghodsi、特斯拉 Data Engine 团队。

#### 三个跃迁点

- **L1 → L3 是"加宽"**:把视野从"一种数据"扩到"所有模态的数据"——这一步主要靠主动学习和动手实践
- **L3 → L5 是"加深"**:从"会处理"升到"会架构"——这一步靠在大规模真实场景中练出来
- **L5 → L7 是"换维度"**:从"工程"跨到"工程 + 算法"——这一步往往要切换岗位、补数学和 ML 知识

数据这个维度,**L7 是 2026 年最值钱的位置**。因为 AI 时代的本质是"数据 → 模型 → 价值"这个链条,**懂数据工程 + 懂 AI 工程的人极少**。纯算法的人不懂数据管线,纯后端的人不懂模型训练。两者交集的人就是大模型时代的稀缺资源。

### 1.4 多模态 RAG 怎么搭

#### 先对齐:RAG 是什么

RAG = Retrieval-Augmented Generation,**检索增强生成**。

最经典的(纯文本)版本:
1. 把你的文档切成小块,每块用 embedding 模型转成向量
2. 存进向量数据库
3. 用户提问时,把问题也转成向量,去库里找最相似的几块
4. 把找到的文档块塞进 prompt,让大模型根据这些内容回答

**核心解决的是"大模型不知道你私有数据"的问题**——通过检索,把相关知识临时塞进上下文。

#### 多模态 RAG 要解决的问题

如果知识库里不只有文字,还有图片、表格、视频、音频呢?

- 一份产品手册里既有文字说明也有结构图
- 一个客服库里有问题描述 + 截图 + 录音
- 一个医疗系统里有病历文本 + CT 影像 + 化验单
- 一个监控系统里有日志 + 摄像头视频 + 报警事件

这就是**多模态 RAG**要解决的问题。三种主流方案:

#### 方案 1:统一向量空间(最优雅、最难做对)

**用一个能同时理解多种模态的 embedding 模型,把所有数据(文字、图片、音频)都映射到同一个向量空间**。这样不管你存的是文字还是图片,**用文字也能搜到图片,用图片也能搜到文字**。这就是 CLIP 模型干的事——它把图像和文本对齐到同一个向量空间。

具体流程:

```
入库阶段:
  文档文字 → CLIP/BGE-M3 文本编码器 → 向量 → 向量库
  图片     → CLIP 图像编码器       → 向量 → 同一个向量库
  音频     → CLAP 音频编码器       → 向量 → 同一个向量库

检索阶段:
  用户问题(文字) → 文本编码器 → 向量 → 向量库找相似 → 返回文字+图片+音频混合结果
  用户拍张照 → 图像编码器 → 向量 → 同样能搜出相关文字和图片
```

**适用场景**:电商搜索(以图搜图、以文搜图)、相册搜索、跨模态检索

**问题**:
- 通用 embedding 模型在专业领域(医学、工业、法律)效果差,经常需要 fine-tune
- 不同模态的"语义对齐"很难做到完美——文字"红色裙子"和一张红裙子图片,向量距离可能远不如两张红裙子图片之间近
- 长文本和单张图片的信息密度差异巨大,放在同一空间会有 bias

#### 方案 2:多模态转文字(最实用、工业界主流)

**先把所有非文字数据"翻译"成文字,然后用最成熟的文本 RAG**。

具体流程:

```
入库阶段(预处理):
  PDF 文档 → 文字直接抽取
  图片     → VLM(GPT-4V / Claude / Qwen-VL)→ 生成详细描述文字
  表格     → VLM 或专门的表格解析模型 → 生成 Markdown 表格 + 描述
  音频     → Whisper → 转成带时间戳的文字
  视频     → 抽帧 + VLM 描述每一帧 + Whisper 转音频 → 合成文字摘要

  所有文字 → 切块 → embedding → 向量库
  同时:原始文件路径 / 视频时间戳 / 图片位置 → 元数据存储

检索阶段:
  用户问题 → 文本搜索 → 找到相关文字块 → 同时取出元数据指向的原始图片/视频
  → 把"文字 + 原始图片/视频片段"一起塞给多模态大模型生成答案
```

**为什么是工业界主流**:
- 复用最成熟的技术栈——文本 RAG 这一套已经非常成熟
- VLM 描述图片这件事现在很便宜很准——Claude / GPT-4V / Qwen-VL 都很强了
- 可解释、可调试——每一步都是文字,出了问题容易定位
- 检索精度可控——文本 embedding 比跨模态 embedding 稳定得多

#### 方案 3:混合方案(高端做法)

实际工业级系统通常**两种方案叠加**:
- **粗筛用统一向量空间**(快、广)
- **精排用方案 2 的"转文字+VLM 理解"**(准、深)
- **再加上传统检索**(BM25 关键词、知识图谱、规则过滤)

这就是所谓的**混合检索(Hybrid Retrieval)**。

#### 工业级架构(2026 年生产环境的真实做法)

```
原始数据
  ↓
预处理管线(VLM + Whisper + OCR + 表格解析)
  ↓
统一的"富文本表示"(文字描述 + 原始资源指针)
  ↓
分块 + 向量化(BGE-M3 / OpenAI / Cohere embedding)
  ↓
向量库(Milvus / Qdrant / Pinecone / 国内 Zilliz)
  ↓
检索(向量召回 + BM25 关键词 + 重排序 Reranker)
  ↓
组装 prompt(文字 chunks + 原始图片/视频片段 base64)
  ↓
多模态大模型生成答案(GPT-4V / Claude / Gemini)
  ↓
返回答案 + 引用源(可点开看原图/原视频)
```

#### 几个真实场景

**场景 1:企业知识库 RAG(最常见)**
- 文档:Word / PDF / Markdown / 网页
- 难点:PDF 里的表格、图表、公式
- 方案:用 Unstructured / LlamaParse / 专门的文档解析工具把 PDF 拆成文字 + 表格 + 图片;表格转成 Markdown 格式;图片用 VLM 生成 caption,存"caption + 图片路径";文字部分正常 chunk + embedding

**场景 2:电商商品搜索**
- 数据:商品标题 + 描述 + 多张图片 + 用户评论
- 方案:商品文字走 BGE-M3 embedding;商品图片走 CLIP embedding(中文场景用 Chinese-CLIP);用户搜"红色连衣裙夏季短款",同时召回文字相似商品 + 图片相似商品;用户上传一张图,反过来用图搜商品

**场景 3:医学影像辅助诊断**
- 数据:病历文字 + CT/MRI 影像 + 化验单
- 方案:专科 VLM(LLaVA-Med、医学专用模型)给影像生成描述;病历用医学专用 embedding(BiomedBERT);用户输入症状 → 检索相似病例(文字) + 相似影像(向量);把"相似病例摘要 + 当前患者影像"给医学 LLM 生成参考意见
- **强人工审核**——医疗场景不能纯自动化

**场景 4:视频监控问答(L7 级别的项目)**
- 数据:几十路摄像头连续录像
- 方案:实时抽帧(每秒 1 帧)→ VLM 生成描述 → 带时间戳存进库;重要事件检测(YOLO 识别人 / 车 / 异常行为)→ 触发更密集的描述;音频走 Whisper 转写;用户问"昨天下午 3 点东门有没有快递员经过"→ 检索相关时段描述 → 调出对应视频片段 → VLM 再确认 → 返回答案 + 视频证据

#### 工程难点和踩坑

**1. Chunking 策略对多模态特别难**——文字按段落切,图片按什么切?表格切了上下文丢了怎么办?现在主流做法是**保留"图文邻接关系"**——一张图旁边的文字、表格的标题和说明,要绑在一起。

**2. Embedding 模型的选择直接决定效果上限**——中文场景:BGE-M3、BCE、Conan-embedding 是 2026 年主流;英文:OpenAI text-embedding-3、Cohere embed-v3、Voyage AI;多模态:CLIP / SigLIP / Chinese-CLIP / Jina-CLIP。**选错了你后面优化什么都救不回来**。

**3. 重排序(Reranker)是隐形的性能杀手**——向量召回的 top 50 里,真正相关的可能只有 5 个,顺序还是错的。**必须再过一道 Reranker**(Cohere Rerank、BGE-Reranker、Jina Reranker)才能拿到生产级的精度。

**4. 多模态上下文窗口压力**——图片转 base64 进 prompt,token 消耗比纯文本高一个量级。GPT-4V 一张高清图能吃 1000+ tokens。**必须做图片压缩、resize、按需加载**。

**5. Cost 控制**——预处理阶段每张图都调 VLM,百万张图就是几万美元。**冷热数据分层、缓存、批量调用**是必须的。

### 1.5 多模态大模型 vs 多模态 RAG —— 别再混淆

**Kimi、千问、GPT-4o、Claude、Gemini 这些是"多模态大模型",不是"多模态 RAG"**。

它们是**模型本身**,RAG 是**用模型搭建的一种系统架构**。这两个是完全不同维度的东西。

打个比方:
- **大模型** = 厨师(有做菜的能力)
- **RAG** = 一种菜谱组织方式(让厨师先去仓库找食材,再做菜)

#### 它们各自是什么

**Kimi(月之暗面)**:大语言模型,以**超长上下文**著称(早期主打 200 万字),后来加入了多模态能力(能看图、读 PDF)。是一个**模型 + 产品**。

**千问 / Qwen(阿里)**:一个完整的模型家族,包括 Qwen-Max(纯文本)、Qwen-VL(视觉)、Qwen-Audio(音频)、Qwen-Omni(全模态)。既有闭源大模型,也开源了一系列模型。

它们是"多模态大模型"——**意思是模型本身能处理多种模态的输入(文字、图片、有时还有音频、视频)**。

#### 它们和 RAG 是什么关系?

它们是 **RAG 系统里的"生成"那一环**。

回到 RAG 的定义:**Retrieval(检索) + Augmented(增强) + Generation(生成)**。
- **Retrieval**:向量数据库 + embedding 模型 + 检索策略——这部分和 Kimi / 千问没关系
- **Augmented**:把检索到的内容塞进 prompt——这部分是工程
- **Generation**:让大模型根据这些内容生成答案——**这一步才用到 Kimi / 千问 / GPT-4o**

所以更准确的说法是:**"我用千问 VL 作为生成模型,搭建了一套多模态 RAG 系统"**——千问只是其中的"大脑",不是整个系统。

#### Kimi 自带的"上传文件问答"算 RAG 吗?

算,但是它是**最简单形态的 RAG**——而且严格说,Kimi 那种因为上下文超长,经常**直接把整个文档塞进上下文,不做检索**,这甚至都不算严格意义上的 RAG,更像是"长上下文直接处理"。

真正的"RAG 系统"特征是:
- 有**自己的知识库**(几万到几亿条数据)
- 有**独立的检索层**(向量库 + 索引)
- 有**工程化的处理管线**(分块、embedding、重排、缓存)
- 大模型只负责"在被检索出来的内容里生成答案"

**Kimi 网页版、千问网页版让你上传几个文件提问,这是它们的"产品功能",不是真正的 RAG 系统**。

#### 真假 RAG 的判断标准

**场景 A:你直接用 Kimi 网页版**
- 你上传一份 200 页的产品手册 → Kimi 把全文塞进它的超长上下文 → 你问问题,它从全文里找答案
- **这是"长上下文 + 多模态模型",不是 RAG**

**场景 B:你给公司搭一个客服系统**
- 公司有 10 万份历史工单 + 5000 张产品图 + 1000 段客服录音
- 这些数据塞不进任何模型的上下文(就算 200 万 token 也不够)
- 你必须搭一套:数据预处理 → 向量库 → 检索 → 重排 → 拿 top 10 结果给大模型
- 大模型可能选千问 VL 或 GPT-4o
- 用户问问题,系统先检索再生成答案
- **这才是真正的多模态 RAG 系统**——Kimi / 千问只是其中的"生成器"角色

## 第二部分:AI 大模型世界 —— 以"Token 流"为锚点

### 2.1 大模型的锚点:Token 流

软件的锚点是调用链(请求怎么流过模块),硬件的锚点是信号链 / 功率链(物理量怎么流过元件)。**大模型的锚点是 Token 流**(数据怎么变成 token、流过模型、再变回输出)。

具体长这样:

```
原始数据(文本/图像/音频/视频)
    ↓ 编码 / Tokenize
Token 序列(模型能"读懂"的最小单位)
    ↓ Embedding 层
向量(每个 token 变成几千维的数字)
    ↓ N 层 Transformer(注意力 + 前馈)
内部表示(每一层都在"理解"得更深)
    ↓ 输出层
下一个 Token 的概率分布
    ↓ 采样
输出 Token
    ↓ 解码
文本 / 图像 / 音频 / 行为
```

**这就是大模型的"链路"**——所有 AI 大模型,本质上都是这条链路。差别只在每一段怎么实现、参数多大、训练数据多大。

理解这条链路,就和理解软件的请求链路、硬件的信号链一样——所有问题都能挂回这条链上。

### 2.2 大模型层级 L1-L9

#### L1 - 会用 API
调通了 OpenAI / Claude / Qwen 的 API,能写一个简单的聊天机器人。
**自测**:能用 Python 写个 10 行代码调通 GPT 吗?
**对标**:**任何会写代码的人,1 天就能到**。

#### L2 - 会做 Prompt Engineering
懂得怎么写 prompt 让模型表现好——few-shot、角色扮演、思维链、输出格式约束、防止幻觉。
**自测**:让模型把一段非结构化文字提取成 JSON,你能让它的成功率从 60% 提到 95% 吗?
**对标**:**初级 AI 应用开发,1-3 个月**。这是 2026 年大部分"AI 工程师"的实际水平。

#### L3 - 会搭 AI 应用系统
能把模型 + 向量数据库 + 工具调用 + 工作流编排成一个完整应用——RAG、Agent、Function Call、MCP。会用 LangChain / Dify / LlamaIndex。
**自测**:能从零搭一个企业知识库问答系统,从文档解析到答案生成端到端跑通吗?
**对标**:**中级 AI 应用工程师,3-6 个月**。**这是 2026 年招聘市场上"AI 工程师"最主流的位置**。

#### L4 - 懂模型推理与工程优化
懂模型在跑的时候发生了什么——KV Cache、批处理、量化、推理加速、并发、延迟、显存、成本。会用 vLLM / SGLang / TensorRT-LLM 做推理优化。
**自测**:同样的 Qwen-72B,怎么让它在一张 A100 上吞吐量提升 10 倍?如何把推理成本砍到 1/5?
**对标**:**AI Infra 工程师,1-2 年**。**国内大厂 P7、字节 3-1**。这一层人极其稀缺,因为 2024 年才真正成为正式岗位。

#### L5 - 会微调和后训练
能根据业务数据微调一个开源模型——SFT(监督微调)、LoRA、DPO、RLHF。懂得怎么准备数据、怎么评估、怎么避免 overfit / catastrophic forgetting。
**自测**:有 1 万条客服对话数据,怎么微调一个 Qwen-7B 让它在你公司的客服场景上比 GPT-4 还好用?
**对标**:**算法工程师,2-4 年**。**国内大厂 P7-P8**。

#### L6 - 懂模型架构与训练
理解 Transformer 的内部机制——Attention 怎么工作、为什么 MoE 能省算力、为什么长上下文要用 RoPE / ALiBi、为什么训练大模型要用流水并行 + 张量并行 + 数据并行。能从零训练一个小规模模型(几亿到几十亿参数)。
**自测**:解释为什么 KV Cache 能加速推理但占显存?DeepSeek 的 MLA 比 MHA 省了什么?MoE 模型推理时只激活部分参数是怎么做到的?
**对标**:**资深算法工程师 / 大模型团队成员,3-6 年**。**国内顶级大厂大模型团队、Anthropic / OpenAI 普通研究员**。

#### L7 - 大模型预训练与对齐
参与从零训练一个百亿到千亿级模型的全流程——数据清洗 / 配比、训练稳定性、scaling law、对齐、RLHF、reward model、安全。能 own 一个完整的预训练或后训练阶段。
**自测**:训练 100B 参数模型时,loss 突然飙升怎么排查?数据混合比例如何决定?为什么 RLHF 容易导致 reward hacking?
**对标**:**头部大模型公司核心成员**(国内 DeepSeek / Moonshot / 阿里通义 / 智谱 / 百川 / 字节 Seed / Anthropic / OpenAI / Google DeepMind 的研究员)。**全球可能就 1-2 万人**,薪资百万到千万级。

#### L8 - 模型创新者
提出新架构、新训练范式、新 scaling law,**让一个新方向被全行业跟着用**。
- 架构创新:Transformer(Vaswani 等)、MoE(Shazeer)、Mamba / State Space Models
- 训练范式:RLHF(Christiano 等)、Constitutional AI(Anthropic)、Test-time Compute(o1 / DeepSeek-R1 团队)
- 效率突破:DeepSeek 的 MLA + GRPO、FlashAttention(Tri Dao)
- 多模态范式:CLIP(Radford)、DALL-E、Diffusion Transformer

**对标**:**顶级实验室的 First Author**。**全球可能几百到几千人**。

#### L9 - 范式定义者
定义"AI 这件事下一步往哪走"的人。这一层不是技术成就,是**改变行业方向**:
- Geoffrey Hinton / Yann LeCun / Yoshua Bengio:深度学习三巨头
- Ilya Sutskever:推动 Scaling Law、GPT 路线
- Dario Amodei / Jared Kaplan:推动了 Scaling Law 论文 + Anthropic 的安全路线
- Demis Hassabis:定义了 DeepMind 的科学+AI 路线(AlphaGo / AlphaFold)
- Jeff Dean:Google 的整个 AI 基础设施奠基人

**对标**:**全球级 AI 领袖**。**全球几十人**。

### 2.3 链路上每段的"通信方式"

回到锚点视角——和软件链路上"每一段怎么通信"是一样的思路。Token 流过模型时,每一段也有它的"通信方式":

| 链路阶段 | 关键技术 / 决策 | 类似软件里的什么 |
|---------|----------------|-----------------|
| **数据 → Token** | Tokenizer(BPE / SentencePiece / Tiktoken) | 序列化 / 编码 |
| **Token → Embedding** | 词嵌入矩阵 | ID 转向量 |
| **Embedding → 注意力** | Self-Attention / Cross-Attention / MoE | 模块间调用 |
| **注意力 → FFN** | 前馈网络 / GLU / SwiGLU | 计算节点 |
| **多层之间** | 残差连接 / LayerNorm | 数据流转 |
| **训练时反向传播** | 梯度 / 优化器 / 学习率 | 反馈回路 |
| **推理时生成** | 采样策略(temperature / top-p / beam search) | 决策机制 |
| **长上下文** | RoPE / ALiBi / Sliding Window / KV Cache | 状态管理 |

**每一个环节都有 L3 级别的"通信方式选择"**——选错了模型就不 work,选对了就比别人快 10 倍 / 准 10 倍。

#### 真实例子 1:为什么 DeepSeek 能用 1/20 的成本达到 GPT-4 水平?

不是模型更聪明,是这条链路上他们做了多个 L3 级别的优化:
- **Attention 那段**:用 MLA(Multi-head Latent Attention)代替标准 MHA,KV Cache 缩小 90%
- **FFN 那段**:用更稀疏的 MoE,只激活 37B / 总 671B 参数
- **训练那段**:FP8 混合精度,显存和带宽都省一半
- **数据那段**:数据配比和质量做得极好,少而精

**这就是 L7-L8 的工作——不是"造更大的脑子",而是"重新设计每一段链路"**。

#### 真实例子 2:为什么长上下文这么难做?

链路上的一个具体瓶颈:Self-Attention 的计算量和上下文长度的平方成正比。100 万 token 的上下文,理论计算量是 1 万 token 的 1 万倍。所以做长上下文,要在链路上动手术:
- 改 Attention 形式(Flash Attention、Ring Attention)
- 改位置编码(RoPE 的频率基底改造)
- 改 KV Cache(分页、卸载、压缩)
- 训练时分阶段(先短后长)

每一个改动都对应链路上某一段的"通信方式"重新设计。

### 2.4 大模型层级和软件、硬件层级的关键区别

#### 1. 层级跨度极大、不连续

软件 / 硬件从 L1 到 L9 是连续的,有清晰的成长路径。**大模型不是**——L1-L4 是工程,L5-L7 是算法,L8-L9 是研究,**三段是断裂的**,跨段几乎要重新学。一个 L4 的 AI Infra 工程师想做 L7 预训练,基本要回到学校或加入大厂研究院从零开始。

#### 2. 时间维度很短

软件 / 硬件的 L7+ 通常需要 10-15 年,**大模型的 L7 很多人 3-5 年就到了**——因为这个领域才爆发了几年,**资深从业者本身就少**。一个 2022 年开始做大模型的博士生,2026 年可能已经在 OpenAI / Anthropic 当 senior researcher 了。

#### 3. L4 是当前最被低估的层级

大家都盯着 L7 / L8 的研究员明星,但 2026 年市场上**最缺的其实是 L4**——懂推理引擎、懂分布式训练、懂硬件特性的"AI Infra 工程师"。因为模型大家都能下载,**真正的差异化在"如何高效跑起来"**。这个方向的薪资国内已经追到了 100-200 万。

#### 4. 工程层(L1-L4)正在被 AI 自己快速吞噬

讽刺的是,L1-L4 这部分(prompt 工程、RAG 搭建、Agent 编排)**正在变得越来越"自动化"**——Cursor、Claude Code 都能帮你写。**所以 L3 这个位置的窗口期可能就 2-3 年**,2028 年之后纯做"会调 API + 搭 RAG"的工程师价值会大幅缩水。要么往上到 L4-L5(算法 + Infra),要么横向走业务深度(垂直行业的 AI 应用)。

#### 5. L8-L9 高度依赖资源

软件领域一个天才工程师在自己宿舍可以写出影响行业的开源项目(Linux、Git)。**大模型领域,你没有 1 万张 H100 / B200 + 几亿训练经费,根本没法做 L8 级别的工作**——这是这个领域的残酷之处。所以 L8+ 基本只在 OpenAI / Anthropic / Google / Meta / DeepSeek / Moonshot 这种少数公司里诞生。

## 第三部分:把数据和 AI 也放进系统思维全景

### 3.1 跨领域的"链路同构表"

把目前讨论过的所有领域放在一起,你会看到一个非常美的对称:

| 领域 | 链路锚点 | 流动的是什么 |
|------|---------|-------------|
| **软件** | 调用链 | 请求 / 数据 |
| **硬件** | 信号链 / 功率链 | 电信号 / 电流 |
| **数据** | 数据管线 | 各种模态的数据 |
| **大模型** | Token 流 | Token / 向量 |
| **机器人** | 感知 - 决策 - 行动链 | 物理世界信号 |
| **生物** | 神经信号链 / 代谢链 | 电化学信号 |
| **经济** | 资金流 | 钱 |
| **组织** | 信息流 / 决策链 | 信息 |

**所有复杂系统都可以用"流"的视角拆开**——这就是系统思维的本质。

### 3.2 给你的具体建议

回到这份文档的起点——"我只关注表格数据"是国内大部分软件工程师的盲区。突破这个盲区的具体路径:

#### 第一步:建立"数据世界地图"在脑子里

不需要懂每一种,但要知道**它存在、大致用在什么场景、对应的技术栈关键词是什么**。这样下次遇到一个新需求,你脑子里能瞬间反应"哦这是时序数据,关键词是 InfluxDB / Prometheus / 异常检测"——你不用马上会做,但你知道往哪查、跟谁合作。

#### 第二步:挑一类深扎下去

按你的兴趣 + 行业前景选一个,深做到 L4-L5:
- 喜欢直观、视觉化的 → **图像 / 视频**
- 喜欢逻辑、抽象的 → **图数据 / 知识图谱**
- 喜欢实时性、工程感强的 → **时序 / 流处理**
- 喜欢和 AI 结合 → **向量数据 / 多模态 / RAG**

#### 第三步:其他领域,保持"知道但不深入"

定期看看新闻、扫一下顶会论文标题、跟一个行业大 V、收藏几个开源项目——**做到"听到不陌生、需要时能上手"就够了**。

#### 一个最小可跑的练手项目:"我的相册问答助手"

如果想真正搞懂多模态 RAG,周末就能做一个:
- 把你电脑里的几百张照片拿出来
- 用 Qwen-VL 或 Claude 给每张图生成一段中文描述
- 描述存进 ChromaDB(本地向量库,5 行代码搭起来)+ 图片路径作为元数据
- 问"我去年在哪个海边拍过日落?"
- 系统检索描述,找到匹配的图,把图 + 描述返回给 LLM,生成自然语言答案 + 显示原图

**用 Cursor 或 Claude Code 一晚上能跑通**。跑通之后,把"照片"换成"PDF"就是企业知识库,换成"商品图"就是电商搜索,换成"病历"就是医疗 RAG——**架构都是一样的**。
