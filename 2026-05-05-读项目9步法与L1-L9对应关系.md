## 三、详细内容

### 3.1 9 步法是什么

这不是"逆向工程"——逆向(Reverse Engineering)在工程领域有特定含义:**没有源码、没有文档,通过反编译、抓包、调试器一步步推断出系统怎么工作**。比如分析一个闭源 App 的协议、破解一个商用软件、研究恶意代码——这些是逆向。

而这个方法是:**有源码、有文档,从入口处往业务深处递进式地阅读**。这个过程的专业名字叫:

- **Code Reading / Code Comprehension**(代码阅读)
- **Top-down Reading**(自顶向下阅读)
- **Code Archaeology**(代码考古——在老项目尤其常用)
- **Onboarding Reading**(入职阅读法)

**这是一个有经验工程师入职新公司、接手新项目的标准做法**。

### 3.2 9 步法每一步详解

#### 步骤 0:tree -L 2 看项目结构

```bash
tree -L 2 -I 'node_modules|__pycache__|.git'
```

或者直接打开 IDE 的文件树,**看 30 秒整体目录结构**。这一步只要 1 分钟,但能帮你建立宏观地图——**知道这个项目大概有几个模块、按什么方式组织(分层架构?DDD?Monorepo?)**。

#### 步骤 1:README 了解项目目标、启动方式和整体功能

这是绝对正确的第一步。**70% 的工程师跳过这一步直接看代码**——他们以为自己在"高效阅读",其实是在抓瞎。

README 给你三个关键信息:
- **目标**:这个项目要解决什么问题(决定了你怎么理解后面的代码)
- **启动方式**:怎么把它跑起来(让你能边读边跑)
- **整体功能**:有哪些主要模块(给你一个心智地图)

**进阶技巧**:先把项目跑起来再读代码。看着日志输出 + 读代码,理解速度快 5 倍。

#### 步骤 2:pyproject.toml / requirements.txt 理解依赖和运行环境

这一步信息量极大,很多人忽略。

依赖文件能告诉你的事:
- **技术栈选型**——用了 FastAPI 还是 Flask?用了 SQLAlchemy 还是直接 psycopg?
- **架构风格**——出现 Celery / Redis 队列 → 有异步任务;出现 LangChain / LlamaIndex → 是 AI 应用
- **复杂度**——50 个依赖 vs 200 个依赖,完全不同量级的项目
- **可疑的依赖**——某些"小众但关键"的库往往是项目的灵魂(比如看到 `dspy` 就知道用了 DSPy 框架做 prompt 优化)

**进阶技巧**:对于不认识的依赖,直接 `pip show` 或者去 PyPI 看一眼介绍——3 分钟就能知道这个库干嘛的,胜过看 1 小时代码瞎猜。

#### 步骤 3:.env / config.py 知道项目需要哪些外部服务

新手通常直接看代码逻辑,**资深工程师都是先看配置**。

为什么?——配置文件是**项目的"接线图"**。它告诉你:
- 项目依赖哪些外部服务(数据库、缓存、消息队列、第三方 API)
- 哪些是必填、哪些是可选
- 不同环境的差异(dev / staging / prod)

看到 `.env` 里有 `OPENAI_API_KEY + REDIS_URL + POSTGRES_URL + NEO4J_URI`,你**不用读一行代码**就知道这是个"用了 AI + 缓存 + 关系数据库 + 图数据库"的项目——大概率是个 AI 应用 / RAG / 知识图谱系统。

#### 步骤 3.5:docker-compose.yml / Dockerfile

如果项目有这个文件,**它是配置的"补充版"**——告诉你:
- 项目实际部署时长什么样
- 各个服务的端口、版本、依赖关系
- 启动顺序(谁依赖谁)

**有时候 docker-compose.yml 比 README 还有信息量**,因为代码不会骗人,但 README 可能过时。

#### 步骤 4:main.py / api/app.py 找到启动入口

这是从"项目元信息"切入"真正的代码"的第一刀。

入口文件能告诉你:
- 程序是怎么启动的(命令行 / Web 服务 / 后台 worker)
- 启动时初始化了什么(数据库连接池、模型加载、中间件注册)
- 路由 / 命令怎么映射到业务逻辑

**进阶技巧**:从入口往下读时,**不要顺着每个函数深挖**——先建立"骨架感",知道程序大致流程,再选一两条主链路深入。否则你会在第一个子函数里就迷路。

#### 步骤 5:数据库连接层 + models

这一步在很多教程里都被跳过——但它至关重要。

为什么?——因为**数据流是一切系统的核心**(链路 + 锚点思维)。

数据库连接层告诉你:
- 用了什么 ORM / Driver
- 是否有连接池、读写分离
- 数据模型(表结构 / 集合结构)
- 查询模式(直接 SQL / ORM / Repository 模式)

**进阶技巧**:数据库连接层之后,**马上看 model / schema 文件**——这些数据结构是项目的"骨骼",看懂它们,你对整个项目的理解就上一个台阶。

#### 步骤 6:主业务链路(RAG / Agent / Tools)

这一步是最有价值的——**理解请求怎么流转 = 理解项目的核心价值**。

但很多人会先看这部分,然后被绕晕。把它放在最后,是对的——前面 5 步是在为这一步做铺垫。读到这里时,你已经知道:
- 项目目标(README)
- 技术栈(依赖)
- 接入的服务(配置)
- 启动方式(入口)
- 数据怎么存(数据层)

**所以读业务代码时,你能"对号入座"——看到一段代码就知道它属于哪个链路、调用了哪个外部服务、产生了哪种数据**。

#### 步骤 7:tests/

很多人完全不看测试,**这是巨大的浪费**。测试是项目最好的"使用说明书":
- 单元测试告诉你**每个函数的预期行为**
- 集成测试告诉你**模块之间怎么协作**
- E2E 测试告诉你**完整的业务场景长什么样**

读完测试,你对项目的理解会再上一个台阶,因为你看到的是**作者认为"这个东西应该这样工作"**——这往往是文档里没写的。

#### 步骤 8:git log

```bash
git log --oneline --graph --all | head -50
git log --stat --since="3 months ago"
```

这能告诉你:
- 项目最近在改什么(说明哪部分是热点)
- 谁在维护(几个人?是否还活跃?)
- 项目的演进历史(从哪里开始,经过几次大重构)

**git log 是项目的"病历"**——从中能看出健康度。如果一个项目 6 个月没更新、最后一次提交是 "fix bug"、commit message 写得乱七八糟——**这个项目大概率不健康,慎用**。

### 3.3 为什么这套流程会和 L1-L9 严密对应?

不是巧合——**因为读项目和理解系统,是同一件事**。

任何复杂系统都可以用"链路 + 锚点 + 层级"拆开。读一个代码项目的过程,本质上就是**还原作者建造这个系统时的思维路径**。作者当初设计系统时是从哪一层开始的?——他先想清楚要解决什么问题(L1)、然后画模块(L2)、再决定通信方式(L3)、再处理故障(L4)……

**你读项目的顺序,就是逆推作者设计的顺序**。

这也是为什么"自顶向下"读代码比"乱翻"快得多——**你在沿着作者的思维链走,而不是在 1000 个细节里大海捞针**。

### 3.4 不同层级的人读同一个项目,看到的东西完全不同

#### L1 级别的人读项目
- 只看 README 的"功能列表"
- 看代码就是"找到那个 if-else 在哪"
- 读完之后:"哦,这个项目能做 XX 和 YY"

#### L2 级别的人读项目
- 能画出基本的模块图
- 看入口文件,知道路由怎么映射
- 读完之后:"哦,前端调 A 接口,A 接口调 B 服务,B 服务读数据库"

#### L3 级别的人读项目
- 能看出"为什么这里用 Redis 做缓存"、"为什么这段用消息队列"
- 看依赖文件就能猜出技术架构
- 读完之后:"这个项目用 RAG + 异步任务 + 双数据库的设计,是因为他们想解决 XX 问题"

#### L4 级别的人读项目
- 能找出代码里所有的"危险点"——重试逻辑在哪?幂等怎么保证?事务边界在哪?
- 读 tests 时能反向理解作者的"焦虑点"
- 读完之后:"这个项目的脆弱点在 X、Y、Z,如果出问题大概率会从这里崩"

#### L5 级别的人读项目
- 看 git log 就能判断这个项目"健康度"
- 能评估这个项目能扛多少 QPS、瓶颈在哪、扩展成本是多少
- 读完之后:"这个项目按现在这个结构,撑到 100 万 DAU 没问题,但到 1000 万会在某某地方崩"

#### L6 级别的人读项目
- 能看出哪些地方是"临时方案",哪些是"长期架构"
- 能预判这个项目 1-2 年后会怎么演化
- 读完之后:"这个项目当前架构是为了快速试错而妥协的,如果业务真起来,他们大概会在 X 时点重构 Y 模块"

#### L7+ 级别的人读项目
- 能从代码组织看出团队结构(康威定律的反向应用)
- 能判断这个项目背后的商业战略
- 读完之后:"这个项目代码风格说明他们团队大概有 3 个 senior + 5 个普通工程师,模块边界对应着他们组织里的 3 个组,他们近期的重心是 XX"

**所以"读完同一个项目能看到什么"——直接反映了你的层级**。

### 3.5 把这套方法变成"修行路径"

这个对应关系可以变成自我提升地图。

你现在能读到 L3——那下一步就是练 L4。具体怎么练?
- **每读一个项目,强迫自己回答 L4 的问题**:这个系统的故障点在哪?重试策略是什么?幂等怎么保证?
- 读不出来,说明项目里没写,**那就是你的练习题——你自己设计一下,如果是你写,会怎么处理**

想练 L5?——读完后回答:这个项目能扛多少 QPS?瓶颈在哪?

想练 L6?——读完后回答:这个项目演化路径是什么?当前是临时方案还是长期架构?

**每个层级,都对应一组"读完项目后能回答的问题"**。能回答几层,你就在第几层。

#### 一个非常实用的自测方法

拿任何一个开源项目(比如 RAGFlow、Dify、LangChain),用 9 步法读一遍。然后问自己:

- 我能读到 L 几?
- 我读到第几步就开始迷糊了?
- 我读完之后能不能用一段话讲清楚这个项目?

**这是最准确的"层级自测题"**——比刷题、比面试、比任何评估都准。代码不会骗人,**你能从代码里"挖出多深的信息",就证明你的认知能挖多深**。

#### 一个具体的输出建议

读完项目后,把你读到的层级用文字写出来。格式可以很简单:

```
项目名:RAGFlow
我用 9 步法读完后,能回答这些问题:

L1:这个项目做企业级 RAG,有文档解析、向量检索、对话 3 大模块
L2:用户上传文档 → DeepDoc 解析 → 切块 + Embedding → 存 Milvus → 检索 → LLM 生成
L3:文档解析用同步 HTTP,Embedding 走异步队列,LLM 调用流式返回
L4:Embedding 失败有重试,但 chunk 处理没看到幂等保证,可能有问题
L5:架构能扛中型企业(几十万文档),百万级文档可能要拆分服务
L6:当前是单体架构,看 commit 像是为了快速验证 PMF 妥协的,未来要拆服务
L7:不知道——我目前到 L6 已经是极限
```

**写出来这个动作,会强制你诚实面对自己的层级**——你写不出来 L4,就说明你 L4 还没到。这比任何"测评工具"都准。

而且发出来还有一个好处:**懂这个项目的人会来评论**。你写错的地方有人指出,你没看到的地方有人补充——**这就是飞轮开始转的瞬间**。

## 四、面试场景的整体回答

> 想象面试官问:"你怎么读一个新的代码项目?"

---

我读一个新项目有自己一套相对固定的流程,大概 9 步,核心思路是**自顶向下、数据流优先**——先建立心智地图,再深入细节,而不是打开 IDE 就到处全局搜关键字。

我会先用 30 秒看一下 `tree -L 2` 的目录结构,知道项目大致由哪些模块组成。然后认真读 README,搞清楚三件事:这个项目要解决什么问题、怎么把它跑起来、整体有哪些功能。**这一步我一定会真的把项目跑起来**——边看日志边读代码,理解速度快 5 倍。

接着我会看 `pyproject.toml` 或 `requirements.txt`。这一步信息量比很多人想象的大——技术栈、架构风格、复杂度都能从依赖里读出来。看到 Celery 就知道有异步任务,看到 LangChain 就知道是 AI 应用,看到 200 个依赖就要小心这是个重型项目。

然后我会去看 `.env` 和 `config.py`。**配置文件是项目的"接线图"**——告诉我项目接了哪些外部服务,哪些是必填、哪些可选。如果有 `docker-compose.yml`,我也会看一下,因为它反映的是真实的部署架构,有时候比 README 还准确。

做完这些"项目元信息"的扫描,我才正式进入代码——从 `main.py` 或 `app.py` 这种入口文件开始。这一步我特别注意**不要顺着每个函数深挖**,先建立骨架感,知道大致流程,再选一两条主链路深入。

之后我会看数据库连接层和 model 定义。**因为数据流是一切系统的核心**,model 是项目的"骨骼",看懂数据结构,整个项目的理解就上一个台阶。

到这一步我才会进入主业务链路——RAG、Agent、Tools 这些核心代码。前面 5 步都是为这一步做铺垫,读到这里时我已经能"对号入座"——看到一段代码就知道它在哪条链路上、调用了哪个外部服务。

最后我会看两个东西:`tests/` 和 `git log`。**测试是项目最好的"使用说明书"**——它告诉我作者认为"这个东西应该这样工作"。**git log 是项目的"病历"**——告诉我项目最近在改什么、谁在维护、是否健康。

---

我之所以用这套流程,**是因为读项目本质上就是用 L1-L9 的视角扫描一个真实系统**:从模块清单(L1)→ 调用关系(L2)→ 通信方式(L3)→ 故障处理(L4)→ 整体健康(L5)→ 演进路径(L6)。**作者当初是按这个顺序设计系统的,我读项目就是在逆推他的思维链**——这比在细节里大海捞针快得多。

**而且这套方法还能反向用作"自我修行"**——我现在能读出 L 几,就证明我目前在 L 几。读不出来 L4 的故障点,就说明我 L4 还没到位,那就是我下一步要练的方向。每读一个项目,都是给系统思维做一次训练。

整个流程一个中等规模的项目大概 3-8 小时能读完,读完之后我对项目的理解,胜过那种只看代码不看文档的人花一个月。
