对。你说的是**真实工程**，那就不能再按“玩具 Demo”讲。

前面那个“不用数据库先跑通”只能算技术验证，不算你要的真实工程。真实工程必须从一开始就按**可运行、可恢复、可扩展、可观测、可持久化**来设计。否则项目名叫“直播 AI 副驾”，实际只是“本地拼装小玩具”，面试官不傻，除非他也只是来凑 KPI。

# 5. 真实工程数据流

## 5.1 创建直播间

```text
主播控制台
   │
   │ POST /api/rooms
   ▼
API Server
   │
   ├─ 生成 room_id
   ├─ 生成 stream_key
   ├─ 生成推流地址
   ├─ 生成播放地址
   └─ 写入 PostgreSQL
   │
   ▼
返回：
{
  room_id,
  rtmp_url,
  flv_url,
  hls_url
}
```

---

## 5.2 主播推流

```text
主播把 rtmp_url 填进 OBS
   │
   ▼
OBS 推流
   │ RTMP
   ▼
SRS
   │
   ├─ 接收直播流
   ├─ 生成 FLV/HLS
   └─ 回调 API Server：房间已开播
```

---

## 5.3 观众看直播

```text
观众打开直播间页面
   │
   │ GET /api/rooms/{room_id}
   ▼
API Server 返回 flv_url/hls_url
   │
   ▼
前端播放器 flv.js/hls.js 播放
   │
   ▼
观众看到直播
```

---

## 5.4 观众发弹幕

```text
观众输入弹幕
   │
   │ WebSocket send
   ▼
Go Gateway
   │
   ├─ 校验 room_id
   ├─ 校验 user_id
   ├─ 生成 message_id
   ├─ 广播给房间所有连接
   ├─ 写 messages 表
   ├─ 转 UnifiedEvent
   └─ 写 Redis Stream
```

---

## 5.5 AI 处理弹幕

```text
Redis Stream: stream:unified_events
   │
   ▼
AI Copilot Worker
   │
   ├─ 读取最近弹幕窗口
   ├─ 构建 LiveSituationState
   ├─ 判断事件价值
   ├─ 决策是否回复
   ├─ 调用 LLM
   ├─ 调用工具
   └─ 写 ai_trajectories
```

---

## 5.6 AI 输出回直播间

```text
AI 决策结果
   │
   ├─ 文字回复 → Go Gateway → WebSocket 广播到直播间
   │
   ├─ TTS 播报 → TTS Service → 主播端播放/OBS播放
   │
   └─ OBS 控场 → OBS WebSocket → 字幕/场景/动画
```

# 6. 真实工程数据库设计

第一版至少这些表。

```text
users
rooms
streams
messages
events
ai_configs
ai_trajectories
ai_feedback
```

## users

```sql
CREATE TABLE users (
  id UUID PRIMARY KEY,
  username VARCHAR(64) NOT NULL,
  role VARCHAR(32) NOT NULL CHECK (role IN ('viewer', 'anchor', 'admin')),
  created_at TIMESTAMP NOT NULL DEFAULT now()
);
```

## rooms

```sql
CREATE TABLE rooms (
  id UUID PRIMARY KEY,
  anchor_id UUID NOT NULL REFERENCES users(id),
  title VARCHAR(128) NOT NULL,
  status VARCHAR(32) NOT NULL CHECK (status IN ('created', 'live', 'offline')),
  created_at TIMESTAMP NOT NULL DEFAULT now(),
  updated_at TIMESTAMP NOT NULL DEFAULT now()
);
```

## streams

```sql
CREATE TABLE streams (
  id UUID PRIMARY KEY,
  room_id UUID NOT NULL REFERENCES rooms(id),
  stream_key VARCHAR(128) NOT NULL UNIQUE,
  rtmp_url TEXT NOT NULL,
  flv_url TEXT NOT NULL,
  hls_url TEXT NOT NULL,
  status VARCHAR(32) NOT NULL CHECK (status IN ('created', 'publishing', 'stopped')),
  started_at TIMESTAMP,
  ended_at TIMESTAMP,
  created_at TIMESTAMP NOT NULL DEFAULT now()
);
```

## messages

```sql
CREATE TABLE messages (
  id UUID PRIMARY KEY,
  room_id UUID NOT NULL REFERENCES rooms(id),
  user_id UUID REFERENCES users(id),
  username VARCHAR(64),
  content TEXT NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT now()
);

CREATE INDEX idx_messages_room_created_at
ON messages(room_id, created_at DESC);
```

## events

```sql
CREATE TABLE events (
  id UUID PRIMARY KEY,
  room_id UUID NOT NULL REFERENCES rooms(id),
  user_id UUID REFERENCES users(id),
  event_type VARCHAR(64) NOT NULL,
  event_json JSONB NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT now()
);

CREATE INDEX idx_events_room_created_at
ON events(room_id, created_at DESC);

CREATE INDEX idx_events_type
ON events(event_type);
```

## ai_configs

```sql
CREATE TABLE ai_configs (
  room_id UUID PRIMARY KEY REFERENCES rooms(id),
  enabled BOOLEAN NOT NULL DEFAULT true,
  tts_enabled BOOLEAN NOT NULL DEFAULT false,
  obs_enabled BOOLEAN NOT NULL DEFAULT false,
  auto_reply_enabled BOOLEAN NOT NULL DEFAULT true,
  max_reply_per_minute INTEGER NOT NULL DEFAULT 5,
  created_at TIMESTAMP NOT NULL DEFAULT now(),
  updated_at TIMESTAMP NOT NULL DEFAULT now()
);
```

## ai_trajectories

```sql
CREATE TABLE ai_trajectories (
  id UUID PRIMARY KEY,
  room_id UUID NOT NULL REFERENCES rooms(id),
  event_id UUID REFERENCES events(id),

  input_event JSONB NOT NULL,
  situation_state JSONB NOT NULL,
  router_result JSONB NOT NULL,
  model_output JSONB,
  tool_calls JSONB,
  tool_results JSONB,
  final_action JSONB,
  latency_ms INTEGER,
  created_at TIMESTAMP NOT NULL DEFAULT now()
);

CREATE INDEX idx_ai_trajectories_room_created_at
ON ai_trajectories(room_id, created_at DESC);
```

## ai_feedback

```sql
CREATE TABLE ai_feedback (
  id UUID PRIMARY KEY,
  trajectory_id UUID NOT NULL REFERENCES ai_trajectories(id),
  feedback_type VARCHAR(64) NOT NULL,
  score INTEGER,
  note TEXT,
  created_at TIMESTAMP NOT NULL DEFAULT now()
);
```

这套表能支撑：

```text
直播间创建
推流管理
弹幕历史
事件追踪
AI 配置
AI 轨迹
后续训练
```

# 7. Redis 设计

```text
Redis
│
├─ room:{room_id}:connections
│   └─ 当前连接用户
│
├─ room:{room_id}:recent_messages
│   └─ 最近 50~200 条弹幕
│
├─ room:{room_id}:state
│   └─ 当前直播状态
│
├─ ai:{room_id}:cooldown
│   └─ AI 回复/播报冷却
│
├─ ai:{room_id}:recent_actions
│   └─ 最近 AI 动作，防止重复
│
└─ stream:unified_events
    └─ AI 消费事件队列
```

Redis 用于：

```text
实时
临时
高频
可丢失后恢复
```

PostgreSQL 用于：

```text
长期
可查询
可复盘
可训练
```

# 13. 扩展性靠什么保证？

## 1. 靠 room_id 做隔离

所有东西都围绕 room_id：

```text
messages.room_id
events.room_id
ai_trajectories.room_id
Redis room:{room_id}:xxx
AI config by room_id
```

这样以后多房间不会推翻架构。

---

## 2. 靠 UnifiedEvent 做事件抽象

现在只有弹幕：

```text
event_type = danmaku
```

以后可以加：

```text
gift
enter_room
like
follow
mic_transcript
video_frame
pk_event
product_click
order_event
```

AI Worker 不用重写主流程，只是新增 handler。

---

## 3. 靠 Tool 协议做动作扩展

现在输出文字：

```text
chat.reply
```

以后加：

```text
tts.speak
obs.show_subtitle
obs.switch_scene
anchor_panel.notify
moderation.block_user
product.show_card
```

AI 决策层只生成 tool_call，执行层负责具体实现。

---

## 4. 靠 Redis Stream 解耦实时链路

WebSocket 不直接调用 AI。

正确做法：

```text
WebSocket Gateway
   │
   ▼
Redis Stream
   │
   ▼
AI Worker
```

这样 AI 挂了，弹幕系统还能跑。

否则：

```text
WebSocket 直接调 AI
```

AI 一慢，全直播间跟着卡。非常人类，非常脆弱。

---

## 5. 靠 PostgreSQL 轨迹表保留训练可能性

一开始不训练也没关系。

但从 V3 开始就记录：

```text
input_event
situation_state
router_result
model_output
tool_calls
tool_results
feedback
```

这样 V7 训练闭环不是空中楼阁。
