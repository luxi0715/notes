## 一、下划线家族

| 写法 | 含义 |
|---|---|
| `name` | 公开 |
| `_name` | 内部用（约定，不强制） |
| `__name` | 防子类冲突（name mangling） |
| `__name__` | Python 官方魔术名 |
| `name_` | 加后缀下划线，避免和关键字冲突（如 `class_`、`type_`） |

例：

```python
def create_widget(class_, type_):    # 不能用 class、type，加 _ 避开
    ...
```

## 二、常量约定

```python
MAX_RETRY = 3              # 全大写 = 常量
API_TIMEOUT = 5
DEFAULT_HOST = "localhost"
```

**约定**：**全大写 + 下划线**表示"常量，不要改"。Python 不强制，但所有人一看就懂。

## 三、命名风格（PEP 8）

| 对象 | 风格 | 例子 |
|---|---|---|
| **类名** | 大驼峰 PascalCase | `UserService`、`OrderManager` |
| **函数 / 方法 / 变量** | 小写下划线 snake_case | `get_user`、`order_count` |
| **常量** | 全大写下划线 UPPER_CASE | `MAX_SIZE` |
| **模块文件名** | 小写下划线 | `user_service.py` |
| **包名（文件夹）** | 全小写，尽量不要下划线 | `utils`、`models` |

这套是 **PEP 8** 规范，Python 官方风格指南。**99% 的项目都遵守**。

## 四、类型注解（type hints）

```python
def get_user(user_id: int) -> User:
    ...

age: int = 25
name: str = "Tom"
users: list[User] = []
```

| 标记 | 含义 |
|---|---|
| `参数: 类型` | 这个参数应该是这种类型 |
| `-> 类型` | 函数返回这种类型 |
| `变量: 类型 = 值` | 变量声明类型 |

**业界几乎所有现代 Python 代码都用**（FastAPI、Pydantic、新项目）。Python 自己**不强制检查**，但 IDE 和工具（mypy、pyright）会检查。

## 五、装饰器（核心）

### 5.1 一句话本质

> **装饰器 = 一个函数，接收另一个函数，返回一个"加强版"的函数。**
>
> 它解决的是"**在不改原函数代码的前提下，给它加新功能**"。

### 5.2 为什么需要它

假设你有 10 个函数，想给每个都加"打日志"功能：

```python
def get_user(): ...
def save_order(): ...
def send_email(): ...
```

- **笨方法**：每个函数里手动加 `print("调用了")`。10 个函数改 10 次，以后想去掉再改 10 次。
- **聪明方法**：写一个装饰器，贴个 `@log` 标签就完事。

### 5.3 装饰器的写法

```python
def log(func):                         # 接收一个函数
    def wrapper(*args, **kwargs):      # 包装一层
        print(f"调用 {func.__name__}")
        result = func(*args, **kwargs) # 调原函数
        print("调用结束")
        return result
    return wrapper                     # 返回包装后的函数


@log
def get_user():
    print("查询用户")


get_user()
```

输出：

```text
调用 get_user
查询用户
调用结束
```

### 5.4 `@log` 语法的本质

```python
@log
def get_user():
    ...

# 完全等价于：

def get_user():
    ...
get_user = log(get_user)    # ← @ 就是这一句的语法糖
```

**`@log` 就是把 `get_user` 喂给 `log`，再用结果替换原来的 `get_user`。**

### 5.5 装饰器解决的 4 个核心问题

```text
1. 不修改原函数，加新功能         （开闭原则）
2. 一份代码，给多个函数复用       （DRY 原则）
3. 把"业务逻辑"和"通用功能"分开    （关注点分离）
4. 让代码看起来像"贴标签"，可读性高  （声明式编程）
```

### 5.6 业界最常见的装饰器场景

| 场景 | 装饰器 | 用途 |
|---|---|---|
| 静态方法 | `@staticmethod` | 不需要 self 的方法 |
| 类方法 | `@classmethod` | 用 cls 而非 self |
| 属性伪装 | `@property` | 把方法伪装成属性 |
| 自动生成类样板 | `@dataclass` | 自动生成 `__init__`、`__repr__` 等 |
| 缓存结果 | `@functools.lru_cache` | 自动缓存函数返回值 |
| 框架路由 | `@app.route("/users")` | Flask / FastAPI 注册路由 |
| 打日志 | `@log`（自定义） | 调用前后打日志 |
| 计时 | `@timer`（自定义） | 测函数耗时 |
| 权限校验 | `@require_login`（自定义） | 验证用户登录 |
| 重试 | `@retry(times=3)`（自定义） | 失败自动重试 |

### 5.7 和"自动调用"的关系

Flask 的 `@app.route`：

```python
@app.route("/users")
def get_users():
    ...
```

它就是装饰器**做注册表**的典型——把 `get_users` 注册到 Flask 的路由表，请求来时框架查表手动调。

**装饰器 = 框架"看起来自动"的关键魔法**。所有"看起来自动"的背后，都是某段框架代码在手动查表手动调。

## 六、Optional / None 约定

```python
def find_user(id: int) -> User | None:
    """找不到返回 None"""
    ...
```

**约定**：返回类型有 `None` = 这个函数可能找不到东西。调用方要判断。

老写法（仍然常见）：

```python
from typing import Optional

def find_user(id: int) -> Optional[User]:
    ...
```

两种写法等价，新代码推荐用 `User | None`（Python 3.10+）。

## 七、特殊参数标记

```python
def func(a, b, *args, **kwargs):
    ...
```

| 标记 | 含义 |
|---|---|
| `*args` | 接收任意多个位置参数（变成元组） |
| `**kwargs` | 接收任意多个关键字参数（变成字典） |
| `*, x` | 强制 x 必须用关键字传 |
| `x, /` | 强制 x 必须用位置传 |

```python
def create(name, *, role):
    ...

create("Tom", role="admin")    # ✅ 正确
create("Tom", "admin")         # ❌ 报错：role 必须用关键字
```

## 八、dunder 变量（模块元信息）

注意：这些是**变量**不是方法。

```python
__name__          # 当前模块名（直接运行时是 "__main__"）
__file__          # 当前文件路径
__doc__           # 文档字符串
__version__       # 项目版本号（约定）
__author__        # 作者（约定）
__all__           # 用 from xxx import * 时导出哪些（约定）
```

例子：

```python
# my_module.py
__version__ = "1.0.0"
__all__ = ["public_func"]   # import * 只导出这个

def public_func(): ...
def _private_func(): ...
```

## 十二、面试口吻

### 问题 1：聊聊 Python 的命名规范

> Python 的命名规范主要遵循 PEP 8。**类名**用大驼峰 `UserService`；**函数、方法、变量**用小写下划线 `get_user`；**常量**全大写 `MAX_SIZE`；**模块文件名**小写下划线 `user_service.py`。
>
> 此外有几套约定符号：单下划线前缀 `_x` 表示内部用；双下划线前缀 `__x` 触发名称改写防子类冲突；前后双下划线 `__x__` 是 Python 官方魔术名；后缀下划线 `class_` 用来避开关键字冲突。
>
> 这些都是**软约定**——Python 不强制，但所有人一看就懂，是工程化代码的基础。

### 问题 2：装饰器是什么，解决什么问题

> 装饰器本质是一个**接收函数、返回函数**的高阶函数，作用是在**不修改原函数代码**的前提下给它加新功能。语法上 `@log` 等价于 `func = log(func)`，是一种语法糖。
>
> 它解决四个核心问题：**不改原函数加功能**（开闭原则）、**通用功能复用**（DRY）、**业务和通用逻辑分离**（关注点分离）、**贴标签式声明**（可读性）。
>
> 业界用得最多的场景包括：`@staticmethod`、`@classmethod`、`@property` 这些 Python 内置装饰器；`@dataclass` 自动生成类样板；`@functools.lru_cache` 自动缓存；以及各种框架的注册装饰器，比如 Flask 的 `@app.route`、FastAPI 的 `@app.get`，本质都是把函数注册到框架内部的表里，请求来时框架查表调用。
>
> 装饰器是 Python 框架"看起来自动"的关键机制——所有看似自动调用的背后，都是装饰器注册 + 框架查表手动调。

```text
Python 的"约定"分两类：
- 强制的（语法规则）：装饰器、类型注解、*args
- 软约定（PEP 8 等）：命名风格、下划线、常量大写

所有这些约定加起来 = 业界默契：
看一眼就知道这是什么、给谁用、怎么用。
99% 的 Python 代码都在这套规则里转。
```
