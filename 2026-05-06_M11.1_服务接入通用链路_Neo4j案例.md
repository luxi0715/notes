## 1. 核心原则：服务接入不是“装个库”

服务接入不是：

```text
装一个 SDK
```

也不是：

```text
uv add neo4j
```

真正的服务接入是：

```text
服务接入
=
服务启动
+
人工验证
+
驱动安装
+
配置管理
+
client 封装
+
生命周期接入
+
测试验证
```

也就是说：

```text
Docker 负责把服务跑起来；
管理页面 / CLI 负责证明服务本身可用；
驱动负责让代码能调用服务；
.env / config.py 负责管理连接参数；
client.py 负责统一封装连接；
app.py lifespan 负责生命周期；
test 负责证明项目代码真的能用。
```

所以：

```text
装驱动只是让代码“有能力连接”；
完整接入才是让项目“稳定使用这个服务”。
```

## 4. Neo4j：每一层到底解决什么问题

### 4.1 服务启动层

```text
docker-compose up -d
```

作用：

```text
启动 Neo4j 容器。
```

这一层解决的是：

```text
Neo4j 服务有没有跑起来。
```

在 docker-compose.yml 里，Neo4j 通常会配置：

```yaml
neo4j:
  image: neo4j:5.14-community
  container_name: legal-agent-neo4j
  ports:
    - "7474:7474"
    - "7687:7687"
  volumes:
    - neo4j_data:/data
    - neo4j_logs:/logs
```

其中：

```text
7474：Neo4j Browser 页面端口
7687：Neo4j Bolt 协议端口，Python Driver 连接用
```

---

### 4.2 人工验证层

人工验证用 Neo4j Browser：

```text
http://localhost:7474
```

登录后执行：

```cypher
RETURN 1 AS hello;
```

如果返回：

```text
hello
1
```

说明：

```text
Neo4j 容器启动成功；
7474 Browser 页面可访问；
7687 Bolt 连接正常；
账号密码可用；
Cypher 能执行。
```

但这只是证明：

```text
人可以通过浏览器访问 Neo4j。
```

还没有证明：

```text
legal-agent 的 Python 后端代码可以访问 Neo4j。
```

---

### 4.3 语言驱动层（Python）

Python 项目需要安装 Neo4j 官方驱动：

```bash
uv add neo4j
```

它解决的是：

```text
Python 能不能通过 Bolt 协议调用 Neo4j。
```

没有驱动，Python 不知道怎么和 Neo4j 通信。

但注意：

```text
uv add neo4j
≠
项目已经接入 Neo4j。
```

它只是让 Python 具备连接能力，不负责连接信息、连接生命周期和测试验证。

---

### 4.4 配置层

`.env` 里需要写：

```env
NEO4J_URI=bolt://localhost:7687
NEO4J_USER=neo4j
NEO4J_PASSWORD=legal_agent_dev_pwd
NEO4J_DATABASE=neo4j
```

这一层解决的是：

```text
Python 项目怎么知道 Neo4j 在哪里？
用什么账号？
用什么密码？
连哪个数据库？
```

其中：

```text
NEO4J_URI=bolt://localhost:7687
```

意思是：

```text
Python 通过本机 7687 端口连接 Neo4j。
```

这个 7687 来自 docker-compose 的端口映射：

```text
宿主机 7687
↓
容器内部 7687
↓
Neo4j Bolt 服务
```

然后在 `config.py` 里统一读取这些配置：

```python
neo4j_uri: str
neo4j_user: str
neo4j_password: str
neo4j_database: str = "neo4j"
```

这样可以避免在业务代码里到处硬编码：

```python
"bolt://localhost:7687"
"neo4j"
"legal_agent_dev_pwd"
```

这种字符串散落各处，后期维护会变成小型灾难。

---

### 4.5 client 封装层

建议新增：

```text
src/legal_agent/db/neo4j_client.py
```

它的作用是：

```text
统一封装 Neo4j driver 的初始化、获取和关闭。
```

典型结构：

```python
from neo4j import AsyncDriver, AsyncGraphDatabase
from legal_agent.core.config import get_settings

_driver: AsyncDriver | None = None


async def init_neo4j() -> AsyncDriver:
    global _driver
    if _driver is not None:
        return _driver

    settings = get_settings()
    _driver = AsyncGraphDatabase.driver(
        settings.neo4j_uri,
        auth=(settings.neo4j_user, settings.neo4j_password),
    )
    await _driver.verify_connectivity()
    return _driver


async def close_neo4j() -> None:
    global _driver
    if _driver is not None:
        await _driver.close()
        _driver = None


def get_neo4j() -> AsyncDriver:
    if _driver is None:
        raise RuntimeError("Neo4j not initialized; call init_neo4j first")
    return _driver
```

这一层解决的是：

```text
不要每个模块都自己创建 Neo4j 连接；
统一由 db/neo4j_client.py 管理连接生命周期。
```

也就是：

```text
init_neo4j：启动时初始化
get_neo4j：业务代码需要时获取 driver
close_neo4j：关闭服务时释放连接
```

---

### 4.6 为什么要 verify_connectivity()

关键代码：

```python
await _driver.verify_connectivity()
```

作用：

```text
启动时强制和 Neo4j 握手，确认真的能连上。
```

它和 Redis 启动时 ping 的思路一样：

```text
启动时就验证连接
↓
失败立刻暴露
↓
不要等到第一次用户请求时才发现数据库连不上
```

这里涉及“懒加载”和“急加载”的取舍。

懒加载：

```text
init() 立即返回
↓
第一次真正查询时才连接
↓
第一次请求可能很慢
↓
连接错误也会延迟暴露
```

急加载：

```text
init() 阶段就强制连接
↓
服务启动时就知道 Neo4j 能不能用
↓
失败直接报错
↓
第一次请求更稳定
```

这里选急加载，因为：

```text
图数据库是核心依赖；
如果 Neo4j 连不上，服务不应该假装启动成功；
启动时失败，比用户请求时失败更容易定位。
```

---

### 4.7 应用生命周期层

后端服务不是运行一个脚本就结束，而是一个长期运行的 FastAPI 应用。

所以 Neo4j 连接应该跟应用生命周期绑定：

```text
应用启动
↓
init_neo4j()
↓
应用运行
↓
业务代码通过 get_neo4j() 查询 Neo4j
↓
应用关闭
↓
close_neo4j()
```

也就是：

```text
启动时初始化连接；
运行时复用连接；
关闭时释放连接。
```

如果不接入 `app.py lifespan`，会出现：

```text
连接什么时候创建，不清楚；
连接什么时候关闭，不清楚；
业务代码可能自己乱建连接；
测试和生产行为不一致。
```

这叫连接生命周期失控。

---

### 4.8 测试层

Browser 测试只能证明：

```text
人手动能连 Neo4j。
```

Python 测试要证明：

```text
项目代码能连 Neo4j。
```

测试至少要验证：

```text
1. init_neo4j() 能成功
2. verify_connectivity() 能通过
3. Python 能执行 RETURN 1 AS hello
4. close_neo4j() 能正常关闭
```

如果 Python 也能拿到：

```text
hello = 1
```

才说明：

```text
Neo4j 已经从“浏览器可用”
升级为
“项目代码可用”。
```

# 9. 面试回复版

## 9.1 通用服务接入版

```text
我理解一个外部服务接入项目，不是简单安装一个 SDK，而是一条完整工程链路。首先要让服务本身跑起来，比如通过 Docker、本地安装或云服务；然后用浏览器、CLI 或 ping 这类方式验证服务本身可用；接着根据项目语言安装对应驱动，比如 Python、Java、Node 的 SDK；之后把连接地址、端口、账号密码等写入配置层，再封装统一的 client，避免业务代码到处创建连接；最后把 client 接入应用生命周期，在启动时初始化连接，关闭时释放连接，并通过测试证明项目代码真的能访问服务。
```

---

## 9.2 Neo4j 项目版

```text
以 Neo4j 为例，我不是只安装 neo4j Python 驱动，而是按服务接入链路处理。Docker 先启动 Neo4j 容器，并暴露 7474 和 7687 端口；Browser 执行 RETURN 1 验证服务本身可用；Python 侧通过 uv add neo4j 安装驱动，通过 .env 和 config.py 管理 NEO4J_URI、NEO4J_USER、NEO4J_PASSWORD 等连接配置；再用 db/neo4j_client.py 统一封装 driver 的 init、get 和 close；最后接入 FastAPI lifespan，在启动时 verify_connectivity，关闭时释放连接，并用 pytest 验证 Python 能执行 Cypher。
```

---

## 9.3 Python / Java 差异版

```text
服务层本身不区分 Python 或 Java。比如 Neo4j 通过 Docker 启动时，它只是一个独立服务，不关心后端语言。真正和语言相关的是代码接入层：Python 项目会通过 uv add neo4j 安装驱动，用 .env / config.py 管理配置，用 neo4j_client.py 封装连接，并接入 FastAPI lifespan；Java 项目则通过 Maven 或 Gradle 添加 neo4j-java-driver，用 application.yml 管理配置，用 Neo4jClient、Repository 或 Service 封装访问逻辑，并通过 Spring Bean 生命周期管理连接。
```

