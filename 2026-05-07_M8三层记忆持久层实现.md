# M8 三层记忆持久层实现

## 0. 核心结论

```text
三种记忆本质上都不是“模型自己记住了”，
而是系统在外部存储里保存历史信息，
然后在下一次请求前重新拼进 messages 上下文。
```

也就是说：

```text
Memory 的真正作用 = 管理上下文
```

M8 的三层记忆，本质是围绕 `messages` 做持久化管理：

```text
推理前：
  读记忆 → 拼上下文 messages → 给大模型

推理后：
  当前对话 → 写入记忆 → 给下次使用
```

---

## 1. M8 记忆整体结构

M8 有四层：

```text
M8 Memory
├── Buffer Memory
│   └── 短期记忆，存最近几轮原始对话，Redis
│
├── Summary Memory
│   └── 摘要记忆，存长对话压缩摘要，PostgreSQL
│
├── Hard Memory
│   └── 长期事实记忆，存用户稳定事实，PostgreSQL
│
└── Memory Manager
    └── 调度层，负责读三层记忆、拼 messages、写回记忆
```

说明：

```text
Buffer Memory 不是 stuff memory。
它是短期滑窗记忆，用来保存最近几轮原始对话。
```

---

## 2. Buffer Memory：短期记忆怎么实现

Buffer Memory 存最近几轮原始对话。

代码文件：

```text
src/legal_agent/memory/buffer_memory.py
```

它用的是：

```text
Redis LIST
```

key 格式是：

```text
buffer:{session_id}
```

每条消息存成 JSON：

```json
{
  "role": "user",
  "content": "我刚被公司辞退了",
  "ts": 1710000000.0
}
```

核心设计：

```text
Redis LIST
├── LPUSH：头插新消息
├── LTRIM：保留最新 N 条
├── LRANGE：读取后反转成时间正序
└── EXPIRE：设置 7 天 TTL
```

核心代码逻辑：

```python
pipe.lpush(key, payload)
pipe.ltrim(key, 0, BUFFER_MAX_ITEMS - 1)
pipe.expire(key, BUFFER_TTL_SECONDS)
await pipe.execute()
```

意思：

```text
1. 新消息写进 Redis List
2. 只保留最新 14 条
3. 设置 7 天过期
```

这里：

```python
BUFFER_MAX_ITEMS = 14
```

也就是：

```text
7 轮对话 = 7 条 user + 7 条 assistant
```

它解决的问题是：

```text
用户说“那怎么办？”
系统能知道“那”指的是前面的问题。
```

## 3. Summary Memory：摘要记忆怎么实现

Summary Memory 存的是：

```text
长对话压缩摘要
```

代码文件：

```text
src/legal_agent/memory/summary_memory.py
```

它用的是：

```text
PostgreSQL 的 session_summaries 表
+
LLM 压缩
```

它不是每轮都压缩，而是：

```text
Buffer 满了以后才触发
```

压缩流程：

```text
1. 读取已有 summary
2. 取 Buffer 里最旧的几条消息
3. 把旧 summary + 新对话片段交给 LLM
4. 生成新的 summary
5. 写回 PostgreSQL
6. 从 Buffer 删除已压缩的旧消息
```

核心函数：

```python
compress_and_update(...)
```

内部大致流程：

```python
old_summary = await get_summary(session_id)
```

然后构造 prompt：

```text
已有摘要 + 新对话片段 → 新摘要
```

再调用大模型：

```python
response = await client.chat.completions.create(...)
```

最后写入数据库：

```python
await upsert_summary(...)
```

它解决的问题是：

```text
对话越来越长，不能无限塞进 messages。
```

所以 Summary Memory 相当于：

```text
把旧上下文压缩成一段摘要，继续保留关键信息。
```

## 4. Hard Memory：长期事实怎么实现

Hard Memory 存的是用户长期稳定事实。

代码文件：

```text
src/legal_agent/memory/hard_memory.py
```

它用的是：

```text
PostgreSQL 的 user_facts 表
```

设计是 KV：

```text
user_id + key + value
```

比如：

```json
{
  "location": "长沙",
  "occupation": "程序员",
  "project": "legal-agent"
}
```

读取时：

```sql
SELECT key, value FROM user_facts WHERE user_id = $1
```

写入时：

```sql
INSERT INTO user_facts (user_id, key, value, confidence)
VALUES ($1, $2, $3, $4)
ON CONFLICT (user_id, key)
DO UPDATE SET
    value = EXCLUDED.value,
    confidence = EXCLUDED.confidence,
    extracted_at = NOW()
```

这表示：

```text
同一个 user_id + key：
  有就更新
  没有就插入
```

Hard Memory 的来源一般是实体抽取：

```text
用户说：“我人在长沙，是程序员”
  ↓
Entity Extractor 抽取
  ↓
location = 长沙
occupation = 程序员
  ↓
写入 user_facts
```

它解决的问题是：

```text
用户长期信息不能每次都靠当前对话猜。
```

## 5. Memory Manager：调度层怎么实现

代码文件：

```text
src/legal_agent/memory/memory_manager.py
```

Memory Manager 不是一种存储，而是调度器。

它暴露两个核心 API：

```text
inject_memory_into_messages
record_turn
```

---

## 5.1 推理前：inject_memory_into_messages

它负责：

```text
读取三层记忆，然后拼成完整 messages
```

流程：

```text
inject_memory_into_messages
├── 读取 Hard Memory
│   └── get_user_facts(user_id)
│
├── 读取 Summary Memory
│   └── get_summary(session_id)
│
├── 拼进 system prompt
│   ├── 用户长期事实
│   └── 历史对话摘要
│
├── 读取 Buffer Memory
│   └── get_buffer(session_id)
│
├── 把 Buffer 里的历史对话追加进 messages
│
└── 最后追加当前 user_message
```

最后生成的 messages 大概是：

```text
messages
├── system：基础 prompt + Hard Memory + Summary Memory
├── user：Buffer 里的历史用户消息
├── assistant：Buffer 里的历史助手回复
└── user：当前用户问题
```

结构示例：

```json
[
  {
    "role": "system",
    "content": "系统提示词 + 用户档案 + 历史对话摘要"
  },
  {
    "role": "user",
    "content": "上一轮用户问题"
  },
  {
    "role": "assistant",
    "content": "上一轮助手回答"
  },
  {
    "role": "user",
    "content": "当前用户问题"
  }
]
```

关键点：

```text
Memory 最终不是直接塞进模型脑子里，
而是变成 messages 的一部分。
```

## 5.2 推理后：record_turn

Agent 回答完以后，调用：

```python
record_turn(...)
```

它负责更新记忆。

流程：

```text
record_turn
├── 当前 user_message 写入 Buffer
├── 当前 assistant_reply 写入 Buffer
├── 异步实体抽取
│   └── 抽取用户事实，写入 Hard Memory
│
└── 检查 Buffer 是否满
    ├── 没满：结束
    └── 满了：
        ├── 取最旧几条消息
        ├── compress_and_update 写入 Summary
        └── trim_oldest_n 从 Buffer 删除旧消息
```

核心写入 Buffer：

```python
await append_to_buffer(session_id, "user", user_message)
await append_to_buffer(session_id, "assistant", assistant_reply)
```

异步抽取实体：

```python
asyncio.create_task(_async_extract_entities(user_id, new_msgs_for_entity))
```

这表示：

```text
主回答先结束
长期事实后台慢慢抽取
```

之后检查 Buffer：

```python
buffer_size = await get_buffer_size(session_id)
if buffer_size >= BUFFER_MAX_ITEMS:
    oldest = await get_oldest_n(session_id, SUMMARY_BATCH_SIZE)
    await compress_and_update(...)
    await trim_oldest_n(session_id, len(oldest))
```

意思是：

```text
Buffer 满了
  ↓
取最旧 6 条
  ↓
压缩成 Summary
  ↓
删掉 Buffer 里的旧消息
```

这就是滑动窗口 + 滚动摘要。

## 6. 三层记忆之间的关系

```text
最近消息
  ↓
Buffer Memory：Redis，保存原文

Buffer 满了
  ↓
Summary Memory：PostgreSQL，保存压缩摘要

对话中抽取出的稳定事实
  ↓
Hard Memory：PostgreSQL，保存用户事实 KV
```

运行流程：

```text
推理前
  ↓
Hard Memory + Summary Memory + Buffer Memory
  ↓
拼成 messages
  ↓
给 ReAct / LLM

推理后
  ↓
当前对话写 Buffer
  ↓
抽取事实写 Hard Memory
  ↓
Buffer 满了压缩进 Summary
```

---

## 7. 为什么这样分层

因为三类数据生命周期不同。

| 记忆 | 数据特点 | 存储 | 原因 |
|---|---|---|---|
| Buffer Memory | 最近原文，高频读写，会过期 | Redis | 快，适合短期滑窗 |
| Summary Memory | 长历史摘要，低频更新，需要持久 | PostgreSQL | 稳，适合持久摘要 |
| Hard Memory | 用户长期事实，需要长期保存 | PostgreSQL | 稳，KV 易扩展 |
| Memory Manager | 不存数据，只编排读写 | Python 代码 | 统一控制记忆注入和更新 |

一句话：

```text
Redis 管短期热上下文；
PostgreSQL 管长期持久信息；
LLM 负责摘要压缩和事实抽取；
Memory Manager 负责调度。
```

## 8. 面试表达版

```text
M8 的三层记忆，本质是围绕 messages 上下文做持久化管理。

Buffer Memory 是短期记忆，用 Redis LIST 存最近几轮原始对话。每个 session 一个 buffer:{session_id}，消息以 JSON 形式保存 role、content 和时间戳。写入时用 LPUSH，LTRIM 只保留最新 14 条，并设置 7 天 TTL。

Summary Memory 是摘要记忆，用 PostgreSQL 的 session_summaries 表保存。当 Buffer 满了以后，Memory Manager 会取最旧的几条消息，结合旧 summary 调用 LLM 做滚动压缩，然后把新 summary 写回 PostgreSQL，并从 Buffer 删除已压缩的旧消息。

Hard Memory 是长期事实记忆，用 PostgreSQL 的 user_facts 表保存 user_id、key、value、confidence。它通过实体抽取模块从对话里提取用户稳定事实，比如地区、职业、偏好，然后 upsert 到数据库。

Memory Manager 是调度层。推理前，它通过 inject_memory_into_messages 读取 Hard Memory、Summary Memory 和 Buffer Memory，并拼成完整 messages 给 ReAct Agent；推理后，它通过 record_turn 把当前 user 和 assistant 消息写入 Buffer，异步抽取事实写入 Hard Memory，并在 Buffer 满时触发 Summary 压缩。
```

## 9. 最短版

```text
M8 三层记忆都是围绕上下文做的。

Buffer Memory：Redis LIST，保存最近几轮原始 messages。

Summary Memory：PostgreSQL，保存长对话压缩摘要。

Hard Memory：PostgreSQL，保存用户长期事实 KV。

Memory Manager：调度层，推理前读三层记忆并拼成 messages，推理后写 Buffer、异步更新 Hard Memory，并在 Buffer 满时压缩进 Summary。
```
