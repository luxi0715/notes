# 系统思维:软件与硬件的链路、锚点与层级

> 整理日期:2026-05-05

---

## 引言

任何复杂系统都可以用三个工具拆开——**锚点、链路、层级**。

软件、硬件、数据、AI、机器人、生物、组织、经济、城市,所有复杂系统都同构:都有"流"、都有"链"、都有"层"。学会一次,通用一辈子。

这份文档分四部分:先讲共通的思维模型,再分别用软件和硬件做完整展开,最后讲怎么把这套思维用到自己的项目里。读完之后,你看待任何工程系统的方式都会不一样——不是看零件,是看"流"。

---

## 第一部分:思维模型 —— 三个核心工具

### 1.1 什么是"锚点"

锚点是任何复杂系统的**入口点**——你从哪里看进去,决定了你能看到什么。

软件的锚点是**调用链**(请求怎么流过模块),硬件的锚点是**信号链 / 功率链**(物理量怎么流过元件)。

锚点的本质是"流"——**流动的东西决定了你看待这个系统的方式**。软件传的是数据,硬件传的是电信号 / 电流 / 电磁波 / 热量。介质不同,但思维方式完全相同。

能自然想到"用流的视角看系统"的人,在任何工程领域都跑得比同龄人快——这其实是**系统思维**的本质。任何复杂系统,不管是软件、硬件、生物、组织、城市、经济,都可以用"流"的视角拆开:有什么在流动、从哪流到哪、每段怎么流、堵了怎么办。

### 1.2 什么是"链路"

链路 = 一条"从哪里来 → 经过谁 → 到哪里去"的路径。

软件架构的本质,就是**"模块之间怎么交换信息"**,不是模块本身长什么样。任何一个稍微复杂点的项目,都会被切成几块,每一块自己内部怎么写,其实不是架构问题,是代码问题。**真正决定这个系统好不好用、好不好改、会不会崩,是这些块之间怎么对话**——谁调谁、用什么方式调、传什么数据、出错了谁负责。

模块是器官,通信是血管——只知道有哪些器官没用,要知道血怎么流的,才知道哪里会堵、哪里会破。如果你能说出"用户下单后,订单服务通过 X 方式通知库存服务,库存服务再用 Y 方式更新数据库,失败了走 Z 重试",那你就是真的懂这个项目;如果只能说"有订单模块、库存模块、用户模块",那基本就是 AI 给你列了个目录,你照着填了代码而已。

### 1.3 什么是"层级"

同一个系统,不同观察者看到不同深度。L1-L9 不是评分,是"视角拉远的过程"。

层级有几个关键性质:

**层级是累积的**——L5 的人必然能做 L1-L4 的事,反过来不行。你不可能跳过 L4 直接做 L7,因为不懂故障处理的人去做组织对齐,只会做出一堆漂亮 PPT 落不了地的架构。

**层级也是飞轮**——每上一层,接触到的资源、机会、判断力都升一档。L2 的人投简历找工作;L4 的人被猎头主动联系;L6 的人被人邀请加入项目;L8 的人定义市场要什么。这不是因为人脉,是因为你能解决的问题层级不同——你的稀缺性是指数级增长的。

**层级越高,越是"减法"而不是"加法"**——L1-L4 是加法:学更多技术、记更多协议、用更多工具;L5-L7 是减法:忽略不重要的、聚焦关键的、识别风险的;L8-L9 是更深的减法:做正确的事,而不是把事做正确。很多人觉得"高级 = 懂得更多"——其实高级 = 懂得什么时候不需要懂。

### 1.4 三个工具如何协同

- **锚点**告诉你**从哪里看**(看流,不看零件)
- **链路**告诉你**看到的是什么**(链是什么形状、经过什么节点)
- **层级**告诉你**看得有多深**(在 L2 看是地图,在 L7 看是组织生态)

三个工具一起,才构成完整的系统思维。

下面我们分别用软件和硬件做完整展开,你会看到这三个工具在不同领域里如何具体落地——但**底层逻辑完全相同**。

## 第二部分:软件世界 —— 以"调用链"为锚点

### 2.1 软件的锚点是什么

软件的锚点是**调用链**——请求从哪来、经过哪些模块、怎么返回。流动的是"数据 / 请求"。

任何一个软件系统,无论看起来多复杂,本质上都是无数条这样的链路在运行:用户点一个按钮,触发一条链;后台跑一个定时任务,触发一条链;一个微服务收到消息,触发一条链。**理解一个系统,就是理解它的所有关键链路**。

### 2.2 软件层级 L1-L9

#### L1 - 模块清单
能列出系统由哪些模块/服务组成。
**自测**:你能说出"我的项目有 A、B、C、D 这几块"吗?
**对标**:**应届~初级,0-1 年**。会用框架搭项目就到这层。

#### L2 - 调用关系
能画出模块之间谁调谁,数据从哪流到哪。
**自测**:用户点一个按钮后,请求经过哪些模块?返回链路是什么?
**对标**:**初级到中级,1-3 年**。能独立完成一个完整功能开发就到这层。大部分国内外包/培训班毕业生卡在这。

#### L3 - 通信方式
能说清每一段调用是同步还是异步、用什么协议(HTTP / gRPC / 消息队列 / WebSocket),并说出**为什么**这么选。
**自测**:为什么订单到支付是同步、订单到通知是异步?如果反过来会怎样?
**对标**:**中级到高级,3-5 年**。能做技术选型、参与架构评审。**国内 BAT 的 P5~P6、Google L4 大概在这层**。

#### L4 - 故障处理与一致性
能预判每个调用环节挂掉后会发生什么,并设计重试、降级、幂等、补偿、事务一致性方案。
**自测**:支付扣款成功但消息队列挂了怎么办?同一笔订单被消费两次怎么办?数据库主从延迟导致读到旧数据怎么办?
**对标**:**高级到资深,5-8 年**。能 own 一个核心系统、写 design doc、带小团队。**BAT 的 P7、Google L5(Senior)**。这一层是真正"架构师"的入门门槛。

#### L5 - 非功能需求
能从性能、扩展性、可观测性、安全、成本五个维度同时评估架构。
**自测**:这个系统能扛多少 QPS?瓶颈在哪?怎么水平扩展?监控指标怎么定?日志/链路追踪怎么打?攻击面在哪?一个月云成本多少、能不能砍一半?
**对标**:**资深/技术专家,8-10 年**。**BAT 的 P8、Google L6(Staff)**。能主导一个业务线的架构、对线上事故负责。

#### L6 - 演进与业务对齐
能根据业务发展路径设计"会长大的架构",知道现在为未来的什么做让步、什么时候该重构、什么时候该忍。
**自测**:如果半年后用户量涨 10 倍、加一条新业务线、合规要求变了,这个架构哪里要改?为什么现在不直接做成那样?
**对标**:**架构师/Principal,10+ 年**。**BAT 的 P9、Google L7**。这层开始要懂业务、懂战略,不只是技术问题。

#### L7 - 组织与生态对齐
能把架构和团队结构、外部依赖、行业标准一起设计(康威定律:系统的结构会复刻团队的结构)。
**自测**:这个架构需要几个团队维护?模块边界和团队边界对不对齐?哪些是自研、哪些用云厂商、哪些用开源?三年后这些选择会不会被供应商锁死?
**对标**:**首席架构师/Distinguished Engineer/CTO**。**BAT 的 P10+、Google L8+**。这层基本不只是技术职业,而是技术+组织+商业的复合决策。

#### L8 - 技术战略与商业对齐
到这层,你已经不是在"做架构",而是在用架构这个工具**为公司挣钱 / 省钱 / 建立壁垒**。我们的核心竞争力是什么?架构应该把资源砸在哪里、哪里可以将就?哪些技术能力要做成"平台",对内复用、对外变成产品卖钱?
**对标**:**CTO / VP Engineering / Google L9 (Distinguished)**。一家几百到几千人公司的技术一号位。

#### L9 - 行业级影响
这一层人极少,基本是定义"下一代架构长什么样"的那些人:发明新范式的人(Werner Vogels 推动云计算和 service-oriented 架构、Jeff Dean 的 MapReduce / TensorFlow / TPU、Linus Torvalds 的 Linux / Git)、在新领域开荒的人(Kubernetes / Kafka / React 的核心团队)、技术 + 商业模式都重写的人(Stripe 的 Patrick Collison、OpenAI 早期架构团队)。
**对标**:**Fellow / Chief Architect / 开源标杆项目核心 maintainer / Google L10+**。这层人全球可能就几百个。

### 2.3 用"外卖下单"完整走一遍 L2-L4

我们用一个具体例子,把 L2、L3、L4 三层的差别一次性看清楚。

#### 第一层:模块(L1 给的目录)

用户模块、餐厅模块、订单模块、支付模块、配送模块、通知模块。

看起来很清楚,但这层信息基本没用,因为它没回答任何关键问题。

#### 第二层:谁调谁(L2 的链路)

用户下单这个动作,会触发一连串调用。把它写成箭头:

用户 App → 订单模块(创建订单)→ 餐厅模块(检查餐厅是否营业、菜品是否有货)→ 支付模块(扣款)→ 配送模块(派单给骑手)→ 通知模块(短信/推送告诉用户和餐厅)

到这一步你已经能看出问题了:这一串如果都是"同步调用"(订单模块站在原地等餐厅模块返回,再等支付模块返回,再等配送模块返回……),那用户点"下单"按钮后,可能要等 5-10 秒才看到结果,而且**任何一个环节挂了,整个下单就失败**。

#### 第三层:用什么方式调(L3 真正的架构决策)

现在重新设计这条链路:

- 订单模块 → 餐厅模块:**同步 HTTP 调用**(必须立刻知道餐厅有没有这个菜,不然没法下单)
- 订单模块 → 支付模块:**同步调用**(必须确认扣款成功才能算下单成功)
- 支付成功后 → 配送模块:**异步消息队列**(把"订单 12345 需要派送"丢到 Kafka,配送模块自己去消费,订单模块不等)
- 支付成功后 → 通知模块:**异步消息队列**(同上,发短信慢一点没关系)
- 配送模块 → 骑手 App:**WebSocket 长连接**(骑手要实时收到新单)

为什么这么设计?因为:用户最在乎的是"我钱扣了没、单下了没",这两步必须同步、必须确认;配送和通知慢几秒用户感知不到,所以异步、解耦,**就算配送系统挂了,用户的订单也已经成功了,后面慢慢补**。

#### 第四层:出错了怎么办(L4 架构师真正赚钱的地方)

支付扣款成功了,但消息队列发不出去,配送模块永远收不到这单怎么办?→ 需要"事务消息"或"本地消息表":先把"要发的消息"写进订单数据库的同一个事务里,有个后台任务定期扫表补发。

骑手 App 断网了 30 秒,这期间派的单丢了怎么办?→ 配送模块要有重试 + 超时转派机制。

同一笔订单消息被消费了两次怎么办?→ 配送模块要做"幂等",用订单 ID 去重。

---

看到差别没?**第一层是 AI 给你的目录,第四层是真正的架构**。AI 现在能做好第一层、勉强做第二层、第三层经常给套模板答案、第四层基本要靠人——因为它要结合你的业务场景(用户能容忍多慢?数据丢了能不能补?预算多少?团队几个人维护?)才能拍板。


### 2.4 通信方式深入展开

#### 2.4.1 后端通信的三个层次

后端通信其实分三个层次,理解了这三层,你就能拆解任何一个真实系统。

**第一层:前端 ↔ 后端(外部通信 / 南北向)**

这是用户的浏览器或 App 和你的服务器之间的通信。

- **绝大部分用 HTTP/HTTPS**(REST API、GraphQL)
- 实时场景用 **WebSocket**(聊天、推送、协作)
- 视频流用 **HLS / DASH / WebRTC**
- 推送用 **APNs / FCM / 厂商通道**

为什么这一层几乎必须用 HTTP 类协议?因为:
- **要穿越互联网**——经过用户的 WiFi、运营商网络、各种防火墙、CDN,只有 HTTP/HTTPS 是处处放行的
- **客户端环境不可控**——浏览器、iOS、Android、各种版本,只有 HTTP 兼容性最好
- **要考虑安全**——HTTPS 是事实标准,所有浏览器都强制要求

**第二层:后端服务之间(内部通信 / 东西向)**

这就是"一个进程给另一个进程发送请求"。这一层选择就多了,因为是**自己内网、环境可控**。

同步调用(必须立刻拿到结果):
- **HTTP / REST**——最常见,简单,但慢、没强类型、序列化开销大
- **gRPC**——Google 出的,基于 HTTP/2 + Protobuf。**比 HTTP 快 5-10 倍、强类型、自动生成代码、支持流式**。微服务内部的事实标准
- **Thrift**——Facebook 的,类似 gRPC,国内字节、小米用得多
- **Dubbo**——阿里的,Java 生态首选,国内大厂常用

异步调用(不用立刻拿结果):
- **消息队列**:Kafka(高吞吐,日志/数据流首选)、RabbitMQ(灵活、业务消息常用)、RocketMQ(阿里出的,事务消息强)、Pulsar(新一代,云原生)
- **事件总线**:NATS、Redis Stream

**第三层:服务和基础设施之间**

这一层很多人忽略,但其实占了后端通信的一大半:

- **服务 ↔ 数据库**:MySQL 协议、PostgreSQL 协议、MongoDB Wire Protocol——都是各自数据库自己的二进制协议
- **服务 ↔ 缓存**:Redis 用 RESP 协议,Memcached 有自己的协议
- **服务 ↔ 对象存储**:S3 / OSS / COS 都是基于 HTTP 的 REST API
- **服务 ↔ 搜索引擎**:Elasticsearch 用 HTTP REST,新版也支持 gRPC
- **服务 ↔ 服务发现**:Consul / Etcd / ZooKeeper 各有自己的协议
- **服务 ↔ 监控**:Prometheus 拉 metrics 用 HTTP、链路追踪用 OpenTelemetry

#### 2.4.2 主要协议的选型逻辑

**为什么对外用 HTTP/REST、对内用 gRPC?**

| 维度 | HTTP/REST | gRPC |
|------|-----------|------|
| 协议 | HTTP/1.1 文本 | HTTP/2 二进制 |
| 数据格式 | JSON(文本,冗余大) | Protobuf(二进制,紧凑) |
| 性能 | 慢 | 快 5-10 倍 |
| 类型安全 | 没有,前后端容易对不上 | 强类型,自动生成代码 |
| 流式传输 | 不支持(或难) | 原生支持双向流 |
| 浏览器支持 | ✅ | ❌(需要 grpc-web 网关) |
| 调试 | 方便(curl / Postman) | 麻烦(要专门工具) |
| 生态 | 极其成熟 | 较新但快速发展 |

所以分工很清晰:
- 对外(给前端、给第三方)用 **HTTP/REST**——浏览器友好、调试方便、生态全
- 对内(后端服务之间)用 **gRPC**——快、强类型、可靠

典型架构:

```
前端 App
  ↓ HTTP/HTTPS (JSON)
API Gateway / BFF (Backend for Frontend)
  ↓ gRPC (Protobuf)
内部微服务集群
  ↕ gRPC / 消息队列
其他微服务
```

API Gateway 这一层就是协议转换器:对外说 HTTP,对内说 gRPC。

**什么时候用 WebSocket(以骑手 App 推送为例)**

骑手 App 需要**实时**收到新订单。"实时"意味着:服务器一有新单,几百毫秒内骑手手机就要响。

问题是,**HTTP 这种最常见的通信方式,天生是"客户端问、服务器答"——客户端不问,服务器没法主动说话**。

要让骑手 App 实时收到推送,有几种办法:

- **轮询(Polling)**:骑手 App 每秒钟问一次服务器"有新单吗?"——99% 的请求都是"没有",纯浪费流量和电量,服务器要扛 10 万骑手每秒一次的轰炸
- **长轮询(Long Polling)**:服务器故意不立刻回,挂着这个连接,直到真有新订单才返回——比轮询好,但每次有消息后都要重新建立连接,有开销
- **Server-Sent Events (SSE)**:服务器主动往客户端推消息——但只能服务器→客户端,**单向**。骑手"接单/拒单"这种反向操作没法走 SSE
- **WebSocket**:**握手一次,然后保持一条永久双向通道,服务器能推、客户端也能发**——一条连接干所有事,延迟在 100ms 以内,服务器开销远低于 HTTP

骑手 App 一打开,和服务器建立一个 WebSocket 连接,然后:
- 有新订单 → 服务器主动推过去 → App 立刻响
- 骑手点"接单" → App 通过同一条连接发回去 → 服务器立刻收到
- 骑手位置每 5 秒上报一次 → 同一条连接走 → 服务器更新地图

**为什么不所有推送都用 WebSocket?**

因为 WebSocket 也有代价:
- **要维持长连接**——服务器同时维持几十万条连接,需要专门优化(Netty / Go 这种支持高并发连接的技术栈)
- **手机切后台、锁屏后,系统会掐连接**——所以用户 App(顾客那侧)的"订单状态推送",不能依赖 WebSocket。这种情况要走**APNs / FCM / 厂商推送**(苹果推送服务、Firebase、华为/小米/OPPO/vivo 的推送通道)
- **跨设备/跨网络不可靠**——切 WiFi、进电梯、地铁里没信号,连接会断。所以业务层一定要做**断线重连 + 消息补偿**

所以更完整的版本是:
- 骑手 App(常开,工作时一直在前台)→ **WebSocket**(实时双向)
- 顾客 App(随时关闭/后台)→ **APNs / FCM / 厂商推送**(系统级推送 + WebSocket 备用)
- 商家 PC 端(浏览器,常开)→ **WebSocket**
- 短信通知(兜底,推送都失败时)→ **短信网关**

**同样是"推送"这一件事,根据接收方的状态不同,要选完全不同的通信方式**。这就是 L3 真正在做的事——**不是记住"推送=WebSocket",而是知道"在什么场景、什么约束下,选什么"**。

#### 2.4.3 视频通信为什么特殊

视频通信和普通业务通信完全不一样,它是后端里最特殊、也最有意思的一块。原因很简单:**视频的数据量比普通业务大几个数量级,而且对延迟、丢包、卡顿的容忍度完全不同**。

**先理解视频为什么特殊**

一个普通的下单请求,数据量可能 1KB。一个高清视频流,**1 秒就要传 1-5MB**——是普通请求的几千倍。而且视频是**连续的**,不是发完一次就完事,是要持续不断地传几分钟到几小时。

这带来两个根本性的问题:
1. **带宽吃不消**——不可能像 JSON 那样裸传,必须**压缩**(这就是编解码,H.264 / H.265 / AV1 干的事)
2. **不能丢太多、也不能等太久**——丢一帧画面用户能看出来卡顿,等 200ms 才显示下一帧用户就觉得不实时

不同的视频场景,对"延迟"的要求差异巨大。这就直接决定了用什么协议。

**视频按场景分四类,每类用的协议完全不同**

**1. 点播(VOD)——B 站、Netflix、YouTube**

特点:视频文件已经存在服务器上,你随时点开看。**延迟不重要**,流畅最重要。

用的协议:**HLS** 或 **DASH**。原理很有意思:服务器把一个完整视频**切成几秒一段的小文件**(每段 2-10 秒),客户端按顺序下载播放。本质上**还是基于 HTTP 的**——每个小切片就是一个 HTTP 请求。

为什么这样设计?
- 走 HTTP 就能用 CDN(全世界的 CDN 都为 HTTP 优化过)
- 切片可以按不同清晰度提前生成多套(360p / 720p / 1080p / 4K),网络好就拉高清的、网络差就拉低清的——**这就是"自适应码率"**
- 用户拖进度条只下载对应位置的切片,不用从头来

**所以点播本质上还是 HTTP**,只不过是"高级形态的 HTTP"。

**2. 直播——抖音直播、淘宝直播、Twitch**

特点:实时产生、实时观看,但**允许 1-5 秒延迟**(因为是单向广播,主播不需要立刻看到观众反应)。

用的协议看延迟要求:
- **HLS / DASH(切片直播)**:延迟 5-30 秒。优点是用 CDN、稳定、扛百万人同时看;缺点是慢
- **HTTP-FLV / RTMP**:延迟 1-3 秒。国内直播以前的标配,实时性比 HLS 好
- **低延迟 HLS(LL-HLS)/ CMAF**:延迟 1-3 秒,新一代标准
- **WebRTC**:延迟 < 500ms。**互动直播**(连麦、PK、电商主播实时回答弹幕)必须用这个

国内主流是这样组合的:
- 主播推流到服务器:**RTMP**
- 服务器转码、分发:同时转出 RTMP / FLV / HLS 多种格式
- 普通观众拉流:**HTTP-FLV** 或 **HLS**(走 CDN,扛百万并发)
- 连麦的观众和主播之间:**WebRTC**(超低延迟)

为什么不全用 WebRTC?因为 WebRTC 是为"双向实时通话"设计的,**走 CDN 困难、扛百万级观众成本极高**。所以只在真正需要互动的少数人之间用。

**3. 实时视频通话——微信视频、Zoom、Google Meet、Facetime**

特点:**双向、超低延迟**(必须 < 400ms,否则两个人会抢话),容忍**画质损失换流畅**。

用的协议:**WebRTC**(基于 UDP)

为什么必须用 WebRTC + UDP?
- **TCP 不行**——TCP 丢包了会重传、阻塞,但视频通话**宁可丢一帧也不能卡住**(丢一帧只是花一下,卡住整个画面就僵了)
- **UDP 允许丢包**——丢了就丢了,后面的包继续传,加上抗丢包算法(FEC、NACK、Jitter Buffer)做补偿
- **WebRTC 把所有能力打包好了**——回声消除、降噪、自动调码率、NAT 穿透,这些都是几十年视频通话经验沉淀下来的

所以视频通话和直播底层逻辑完全不同——直播容忍 1-3 秒延迟换稳定,通话容忍画质换实时。

**4. 视频监控 / IoT 视频——海康、大华、家用摄像头**

特点:**长时间持续推流**,需要低带宽、低功耗、设备性能差。

用的协议:**RTSP**(传统监控行业标准)、**ONVIF**(标准化协议)、新一代用 **WebRTC**

家用摄像头(小米、萤石)往往是:摄像头 → 自家云服务器(走自研协议)→ 用户手机 App(WebRTC 或自研)。

**视频协议家族的全景图**

| 协议 | 用在哪一段 | 延迟 | 主要场景 |
|------|----------|------|----------|
| **RTMP** | 推流(主播→服务器) | 1-3 秒 | 直播推流标准 |
| **HTTP-FLV** | 拉流(服务器→观众) | 1-3 秒 | 国内直播观看主流 |
| **HLS** | 拉流(切片) | 5-30 秒 | 点播 + 普通直播 |
| **DASH** | 拉流(切片) | 5-30 秒 | 国际点播 + 直播 |
| **WebRTC** | 端到端实时 | <500ms | 视频通话、互动直播、连麦 |
| **RTSP** | 摄像头到服务器 | <1 秒 | 安防监控 |
| **SRT** | 专业广电传输 | <1 秒 | 媒体行业 |

**视频通信的特殊技术栈**

视频不只是"协议不同",**整个技术栈都是另一套**:
- **编解码层**:H.264(最普及)、H.265(更省 50% 带宽但要授权费)、AV1(免费 + 更高效,YouTube/Netflix 在推)、VP9(Google)
- **传输层**:UDP 为主(用 TCP 是例外)
- **网络层**:专门的 **CDN**、专门的 **音视频云**(腾讯云 TRTC、阿里云 RTC、声网 Agora)
- **基础设施**:**SFU**(选择性转发)和 **MCU**(混合)架构,这是多人视频通话的核心
- **服务端处理**:转码、转封装、水印、AI 鉴黄、字幕生成——这都是独立的处理管线

**视频通信和普通业务通信,差别有多大?**
- 普通业务:**HTTP / gRPC,KB 级数据,毫秒级延迟,失败重试**
- 视频:**专用协议,GB 级数据,精确控制延迟和丢包,失败容忍**

整套技术栈、人才、基础设施都是分开的——做视频的工程师一般不做普通后端,做普通后端的工程师也基本不懂视频。这是后端里少有的、和"音视频算法 + 网络协议 + 硬件编码"强绑定的领域,门槛高、人才少、薪资也高。


### 2.5 一个用户下单的完整通信链(7 种协议同时存在)

把前面所有概念串起来,看一个真实的下单流程涉及多少种协议:

1. 用户 App → API Gateway:**HTTPS + JSON**(外部)
2. API Gateway → 订单服务:**gRPC**(内部)
3. 订单服务 → MySQL 主库:**MySQL 二进制协议**(基础设施)
4. 订单服务 → Redis:**RESP 协议**(缓存库存)
5. 订单服务 → 餐厅服务:**gRPC**(内部同步,要立刻确认)
6. 订单服务 → 支付服务:**gRPC**(内部同步,要立刻扣款)
7. 支付成功 → Kafka:**Kafka 协议**(异步广播)
8. Kafka → 配送服务:**Kafka 协议**(消费消息)
9. 配送服务 → 骑手 App:**WebSocket**(实时推送)
10. Kafka → 通知服务:**Kafka 协议**
11. 通知服务 → APNs / FCM:**HTTPS + 厂商协议**(给用户 App 推送)
12. 通知服务 → 短信网关:**HTTPS**(发短信)

**一个下单动作,涉及至少 7 种不同的通信协议**。每种协议都是为它特定的场景优化的——没有一种协议能搞定所有事。

作为架构师怎么选?记住几条原则:

1. **对外用 HTTP/HTTPS**——没得选,生态决定的
2. **对内同步用 gRPC**——除非团队 Java 栈深、用 Dubbo
3. **解耦 / 削峰 / 广播用消息队列**——Kafka 是默认,事务场景用 RocketMQ
4. **实时双向用 WebSocket**——客户端常驻;非常驻的实时通知用厂商推送
5. **基础设施(数据库 / 缓存 / 存储)**——用人家自己的协议,这个没的选

## 第三部分:硬件世界 —— 以"信号链 / 功率链"为锚点

### 3.1 硬件的锚点是什么

软件的锚点是调用链(请求怎么流过模块),硬件的锚点是**信号链 / 功率链**(物理量怎么流过元件)。和软件的逻辑一模一样,只是介质不同——软件传的是数据,硬件传的是电信号 / 电流 / 电磁波 / 热量。

但和软件不同的一点是:**硬件不是单一一条链,是六条并行的物理链**。

### 3.2 硬件的六条链路

任何一个电子产品,本质上都是几条**链**的组合:

- **信号链**:传感器 → 放大 → 滤波 → ADC → 处理器 → DAC → 输出 (信息怎么从物理世界进系统、再出来)
- **功率链**:电源输入 → 整流 → 变压 → 稳压 → 各路负载 (电怎么从插头流到芯片)
- **时钟链**:晶振 → PLL → 分频 → 各模块 (节拍怎么分发)
- **数据链**:CPU ↔ 内存 ↔ 存储 ↔ 外设 (高速数字信号怎么走)
- **散热链**:芯片发热 → 焊盘 → PCB 铜箔 → 散热片 → 风扇 / 外壳 → 空气 (热量怎么散出去)
- **机械链**:外壳受力 → 螺丝 → PCB → 元器件 → 焊点 (跌落 / 振动时力怎么传)

每一条链都有"起点 → 中间节点 → 终点",和软件链路完全是同构的。每一条链上,都要回答和软件 L3 一样的问题:**这一段是什么形式?用什么标准?出问题怎么办?**

举几个具体例子让你看懂这种同构:

**信号链举例(麦克风录音)**:
麦克风 → 前置放大 → 抗混叠滤波器 → ADC → DSP → I2S → 主控

每一段都要决定:
- 这段是模拟信号还是数字信号?
- 阻抗多少?匹配方式?
- 单端还是差分?
- 频率多高?需要什么级别的走线规则?
- 用 SPI 还是 I2C 还是 UART 还是 I2S?为什么?

**数据链举例(CPU 到内存)**:
- 用 DDR4 还是 LPDDR5?(选"协议")
- 1600MHz 还是 3200MHz?(选"速率")
- 走线要等长 ±5mil、阻抗 40Ω 单端 / 80Ω 差分(选"规格")
- 这一段挂了系统就死,所以信号完整性裕量要留多少?(可靠性预算)

这就是硬件的 L3——和软件 L3"为什么这段同步、那段异步"是一模一样的思维。

**故障应对(L4)**:
软件 L4 问"这段调用挂了怎么办",硬件 L4 问"这段信号失真了怎么办、这路电源跌了怎么办、这条热通路堵了怎么办"。

具体例子:
- 功率链上,负载突然拉满,电源能不能跟上?跟不上会怎样?(瞬态响应)
- 信号链上,ADC 前面的运放饱和了,后面会读到什么?(限幅设计)
- 时钟链上,主晶振挂了,系统能不能切到备用时钟?(冗余)
- 散热链上,风扇坏了,芯片会烧吗?有没有过温降频?(降级策略)

**软件叫"重试 / 降级 / 熔断",硬件叫"裕量 / 保护 / 降频 / 备份"——本质完全一样**。

### 3.3 硬件层级 L1-L9

我以**电子硬件**(消费电子 / 嵌入式 / 通信设备这一类)为主,芯片设计另说。

#### L1 - 元器件认知
能看懂原理图上的符号,知道电阻、电容、电感、二极管、三极管、MOS 管、运放各是干嘛的。
**自测**:看到一个电路图,能不能说出每个元件的作用?
**对标**:**电子工程在校生 / 应届**。课程学完就到这层。

#### L2 - 电路功能模块
能把一堆元件组合起来理解成功能模块——这是电源、那是放大器、这是滤波、那是 MCU 最小系统。
**自测**:看到一块开发板,能不能划出"这块是 XXX 功能"?
**对标**:**初级硬件工程师,0-2 年**。能照着 datasheet 抄电路、跟着别人画板子。

#### L3 - 电路设计与选型
能根据需求自己设计电路:选什么芯片、外围电路怎么搭、为什么用这个拓扑而不用那个。
**自测**:产品要 5V 输入、3.3V/2A 输出、要省电——你选 LDO 还是 DC-DC?为什么?用同步还是异步?开关频率定多少?
**对标**:**中级硬件工程师,2-4 年**。能独立完成一块板子的设计,但量产可能还会出问题。

#### L4 - PCB Layout 与信号完整性
知道电路图画对不等于产品能用。会考虑走线长度、阻抗匹配、回流路径、地分割、串扰、电源完整性、热设计。
**自测**:DDR 走线为什么要等长?为什么差分对要平行紧耦合?GND 平面为什么不能随便割?高速信号下面跨分割会怎样?
**对标**:**高级硬件工程师,4-6 年**。开始能搞定中高速电路(USB、以太网、DDR3 这个量级)。**这一层是硬件工程师真正"分水岭"**——卡在 L3 的人和过了 L4 的人,薪资能差一倍。

#### L5 - 可靠性与 EMC
能预判产品在真实环境里会怎么坏。考虑温度漂移、ESD(静电)、EMI(电磁干扰)、雷击浪涌、跌落、振动、长期老化、批次差异。
**自测**:产品过不了 ESD 8kV 接触放电怎么办?夏天高温下死机是什么原因?为什么实验室好好的,客户现场就是有问题?
**对标**:**资深硬件工程师,6-10 年**。能 own 一个产品从设计到量产、对售后故障率负责。**国内大厂硬件 P7、华为 16-17 级**。

#### L6 - 量产与制造工艺(DFM/DFT)
设计的时候就要考虑工厂能不能造出来、造得便宜不便宜、测试快不快、良率高不高。
**自测**:这个 BGA 焊盘间距 SMT 厂能贴吗?这个测试点位置探针够得到吗?这个螺丝拧紧的力矩工人能稳定控制吗?这个元件是不是只有一家供应商,断供了怎么办?
**对标**:**资深 / 硬件专家,8-12 年**。能主导从样机到百万级量产的全流程。这一层开始要泡工厂、和 PE(制程工程师)/ ME(机构工程师)/ 供应商频繁打交道。**华为 18-19 级、苹果 ICT5**。

#### L7 - 系统架构与跨学科协同
不再只看一块板子,而是看整个产品:硬件 + 结构 + 软件 + 算法 + 散热 + 电池 + 射频 + 工业设计怎么互相妥协。
**自测**:为了让结构再薄 1mm,主板要怎么改?为了多一小时续航,处理器降频还是电池增大?天线性能不够,是动 PCB 还是动外壳?BOM 成本要砍 10 美金,从哪里砍最不影响体验?
**对标**:**首席硬件工程师 / 硬件架构师,10-15 年**。手机、智能手表、汽车这种复杂产品的硬件总负责人。**华为 20 级 +、苹果 ICT6+**。

#### L8 - 平台化与产品矩阵
不再做"一款产品",而是做"一个平台 / 一个产品系列"——共用主板、共用模组、共用供应链,不同档位差异化但研发成本最低。
**自测**:同一颗 SoC 怎么做出旗舰 / 中端 / 低端三个产品?共用的模块怎么切?哪些零件按家族化设计?新一代平台怎么向后兼容三代产品?这个平台决策影响未来 3 年公司几亿营收。
**对标**:**硬件总监 / 平台架构师**。手机厂商、汽车厂商里负责"芯片选型 + 平台规划"的那一小撮人。

#### L9 - 行业级影响
定义"未来 10 年硬件长什么样"的人:
- 推动新工艺 / 新封装 / 新接口 成为行业标准的人(USB-C、PCIe、Chiplet、HBM 背后的核心团队)
- 把一个新形态从概念做到量产并改变行业的人(iPhone 初代硬件团队、特斯拉的 4680 电池团队、Apple Silicon 的 M1 团队)
- 顶尖芯片公司的 chief architect(ARM 的 ISA 设计、NVIDIA 的 GPU 架构、Apple 的 SoC 团队负责人)
**对标**:**Apple Distinguished Engineer / NVIDIA Fellow / 国内华为 Fellow / 几个超级独角兽的 CTO**。全球可能就几百到上千人。

### 3.4 硬件层级与软件层级的对照

把两套层级放在一起,你会发现一个非常美的对称:

| 层级 | 软件 | 硬件 |
|------|------|------|
| **L1** | 模块清单 | 元器件认知 |
| **L2** | 调用关系(链路本身) | 电路功能模块(链路本身) |
| **L3** | 通信方式(每段怎么说话) | 电路设计与选型(每段什么形式) |
| **L4** | 故障处理与一致性 | PCB Layout 与信号完整性 |
| **L5** | 非功能需求(整体体质) | 可靠性与 EMC(整体体质) |
| **L6** | 演进与业务对齐 | 量产与制造工艺 |
| **L7** | 组织与生态对齐 | 系统架构与跨学科协同 |
| **L8** | 技术战略与商业对齐 | 平台化与产品矩阵 |
| **L9** | 行业级影响 | 行业级影响 |

**关键差异**:

- **L4 之前:硬件比软件难**。因为软件 L1-L3 写代码就能验证,硬件 L1-L3 要画板、打样、焊接、测试,反馈周期慢一个数量级。新人成长慢得多。
- **L5 之后:硬件比软件值钱**。因为隐性知识 + 物理世界经验 + 量产 know-how,这些东西**真的没法被 AI 短期取代**,而且培养一个 L5+ 的硬件工程师要 8-10 年,人才供给极慢。
- **天花板上:软件人多、硬件人少**。L7+ 软件架构师全球可能几十万人,L7+ 硬件架构师全球可能几万人——但反过来,**机会也少**,你想做出 iPhone 级别的硬件,全世界没几个公司给你这个舞台。

### 3.5 硬件工程师都在做什么 —— 行业全景

**一句话:你身边所有"通电的东西",背后都有硬件工程师**。但具体分工差别极大,按行业拆一下:

**消费电子**——手机、笔记本、耳机、智能手表、扫地机器人、电视、相机。这是最大的一块,中国是全球产业中心。这里的硬件工程师做的事:选 SoC、设计主板、解决射频天线、搞定续航、过认证、推量产。代表公司:苹果、华为、小米、大疆、Anker、漫步者、影石、Insta360。

**汽车电子**——这两年最火的方向。电动车一台车上有 50-100 个 ECU(电子控制单元),从电池管理、电机驱动、自动驾驶域控、座舱、激光雷达、毫米波雷达、车机一直到车灯,全是硬件工程师在做。代表公司:比亚迪、特斯拉、蔚小理、华为车 BU、博世、地平线、禾赛。**待遇这两年是行业天花板**。

**通信设备**——基站、路由器、交换机、光模块、卫星通信。这里偏射频和高速数字,门槛极高,但也是最稳的赛道之一。代表公司:华为、中兴、思科、爱立信、新华三。

**工业与能源**——电源、储能、光伏逆变器、工业机器人、机床数控、电网设备、医疗仪器。这些产品单价高、寿命长、可靠性要求变态,但**最不容易被取代**,因为知识极度专精、迭代慢、隐性经验多。代表公司:阳光电源、汇川、迈瑞、联影、施耐德、ABB。

**芯片设计**——这是硬件里最"软"的方向,坐在电脑前用 Verilog 写代码、跑仿真、做验证,基本不碰焊台。但工资也是硬件里最高的。分两类:**数字 IC**(SoC、CPU、GPU、AI 芯片)和**模拟 IC**(电源管理、射频、ADC/DAC)。模拟 IC 工程师全球都缺,因为太难培养。代表公司:海思、紫光展锐、地平线、寒武纪、韦尔、卓胜微、汇顶,国外的高通、英伟达、博通、TI。

**机器人 / 具身智能**——刚开始爆发。电机驱动、传感器融合、控制板、电池系统都是硬件工程师的活。代表公司:宇树、傅利叶、智元、特斯拉 Optimus、Figure、1X。

**新兴方向**——AR/VR(Apple Vision Pro、Meta Quest、Pico)、低空经济(无人机、eVTOL)、脑机接口(Neuralink、强脑科技)、量子计算硬件。这些方向人少、机会多,但风险也大。

**他们具体每天在干什么?** 简单说就是这几件事循环:

看需求文档(产品要做什么)→ 选芯片和元件(BOM)→ 画原理图(电路怎么连)→ 画 PCB(物理上怎么布线)→ 打样回来焊接(EVT 阶段)→ 上电、调试、量测(示波器 / 信号源 / 频谱仪 / 网络分析仪)→ 发现问题、改板、再打样(DVT 阶段)→ 解决可靠性、EMC、温度等问题(PVT 阶段)→ 进工厂量产、追良率(MP 阶段)→ 改 cost down 版本继续卖。

一个产品从立项到上市,硬件工程师通常要泡 1-3 年,期间至少打 5-10 版样、跑工厂十几次、和供应商扯皮无数次。

**为什么世界需要这些人?**

因为软件的所有抽象,**最终都要落到一块电路板上才能运行**。GPT 再聪明,也得跑在英伟达的 GPU 上、装在数据中心的服务器里、靠电力公司的电网供电、用台积电的工艺制造、通过华为和思科的网络设备传到你手机上、显示在三星的屏幕上、电池由宁德时代提供、外壳由富士康组装。**这条物理链上每一环都是硬件工程师在撑着**——没有他们,数字世界一秒钟都跑不起来。

### 3.6 硬件 vs 软件:AI 时代的不同命运

把层级图叠上 AI 时代的影响来看就清楚了:

**软件**:
- L1-L3:AI 已经能做不少了。Cursor / Claude Code 这类工具已经能写出大部分典型代码。这些岗位**会被压缩**。
- L4-L5:AI 能辅助但替代不了。架构权衡、系统设计需要结合具体上下文。
- L6+:涉及业务、组织、战略,AI 短期碰不到。

**硬件**:
- L1-L3:AI 已经能做不少了。Cadence、Altium 这些 EDA 工具已经在塞 AI 助手,自动布线、自动选型、自动生成原理图都在快速进化。这些岗位**会被压缩**。
- L4-L5:AI 能辅助但替代不了。仿真、SI/PI 分析这些 AI 能加速,但**现场调试、EMC 整改、可靠性问题排查这些"拿示波器在实验室蹲三天"的事,AI 没有手、做不了**。
- L6+:基本不可能被 AI 替代。涉及到工厂、供应链、跨部门协调、商业判断的工作,AI 连入口都摸不到。

**所以硬件工程师"难被替代"是真的,但前提是你能爬到 L4 以上**。卡在 L1-L3 的硬件工程师,被 AI 替代的速度其实和卡在 L1-L3 的软件工程师差不多——别忘了,**任何行业里"刚入门"的位置都是最容易被替代的**,这一点和具体行业关系不大。

## 第四部分:把思维模型用起来

### 4.1 三个核心认知

**1. 所有复杂系统都可以用"流"的视角拆开**

不管是软件、硬件、数据、AI、机器人、生物、组织、经济、城市,本质都是一样的:有什么在流动、从哪流到哪、每段怎么流、堵了怎么办。能自然想到这一层的人,在任何工程领域都跑得比同龄人快。

**2. 层级越高,讨论的越不像"技术",越接近"判断 + 资源 + 战略"**

不是不提技术,是技术从"决策的主角"变成了"决策的工具"。

| 层级 | 技术在思考中的位置 |
|------|------------------|
| L1-L2 | 技术 = 一切。"我会 Spring Boot 吗?这个 SQL 怎么写?" |
| L3 | 技术 = 选项。"用 HTTP 还是 gRPC?用 MySQL 还是 Postgres?" |
| L4 | 技术 = 权衡。"用这个技术,代价是什么?出问题怎么办?" |
| L5 | 技术 = 资源。"我们有多少机器、多少人、多少钱、多少时间?这些约束下技术选什么?" |
| L6 | 技术 = 路径。"3 年后我们要去哪,现在的技术选型怎么铺路?" |
| L7 | 技术 = 团队的延伸。"我们 hire 得到这种人吗?组织能不能撑住这个架构?" |
| L8 | 技术 = 商业杠杆。"用什么技术能让公司赚到钱 / 建立壁垒?" |
| L9 | 技术 = 行业方向。"下一代技术应该长什么样,我推动它发生。" |

技术不是消失了,是它在思考链条里的位置变了。L1-L4 的考试是"会不会用",L5-L9 的考试是"该不该用、什么时候用、用了之后怎么办、不用又怎么办"。

**3. 找到自己当前所在的层级 + 知道下一层是什么,远比追逐天花板更重要**

绝大多数工程师职业生涯停在 L4-L5 是完全正常的,而且收入也已经很好了。L6+ 不是更"高级"的意思,是岗位性质完全变了——L6 之后越来越不是"写代码"的工作,而是"做决策"的工作,需要的素质也不太一样(沟通、判断、抗压、商业 sense)。

**所以别用层级当焦虑源,用它当地图就行**——知道自己在哪、知道下一步该看什么。

### 4.2 怎么用这套思维诊断自己的项目

最简单的三步法:

1. **画出主链路(锚点 + L2)**——请求从哪进来、经过哪些模块、到哪里去
2. **在每个箭头上标通信方式(L3)**——同步还是异步?用什么协议?为什么?
3. **在每个箭头旁边写"如果这里挂了会怎样"(L4)**——失败模式、重试策略、降级方案

**画三遍,你对自己项目的理解和对架构的理解会完全不一样**。

如果一开始画不出来,说明你还在 L1 看自己的项目——这没关系,**第一次画完整,你就从 L1 跳到 L2 了**。继续追问"为什么这段同步?为什么这段异步?",你就在往 L3 走。再追问"挂了怎么办?",你就在往 L4 走。

层级不是别人评出来的,**是你自己问出来的**。问题的深度决定你看见的深度。

### 4.3 这套思维的可迁移性

软件和硬件只是开头。同样的思维模型,可以用到任何复杂系统:

| 领域 | 链路锚点 | 流动的是什么 |
|------|---------|-------------|
| **软件** | 调用链 | 请求 / 数据 |
| **硬件** | 信号链 / 功率链 | 电信号 / 电流 |
| **数据** | 数据管线 | 各种模态的数据 |
| **大模型** | Token 流 | Token / 向量 |
| **机器人** | 感知 - 决策 - 行动链 | 物理世界信号 |
| **生物** | 神经信号链 / 代谢链 | 电化学信号 |
| **经济** | 资金流 | 钱 |
| **组织** | 信息流 / 决策链 | 信息 |

**所有复杂系统都同构**:都有"流"、都有"链"、都有"层"。掌握一次,通用一辈子。

下次遇到一个陌生领域,不要慌——先问三个问题:

1. **这里有几条链?**(能识别出来的所有"流")
2. **每条链经过哪些节点?**(L2)
3. **每段是什么形式、出问题会怎样?**(L3 + L4)

这三个问题问下去,你就从"看零件"升级到"看系统"了。任何陌生领域,你都能比别人快 10 倍切入核心。
